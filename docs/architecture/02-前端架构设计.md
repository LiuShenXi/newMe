# 前端架构设计

- 日期：2026-04-29
- 技术栈：React Native + Expo + TypeScript + Zustand + React Query
- 适用范围：移动端 App MVP

## 1. 前端架构目标

前端是本产品体验成败的主战场，需要同时满足：

1. 每日记录路径足够短，用户能快速完成今日清单查看和能量记录。
2. 冷启动三路径（深度愿景、快速规划、手动 OKR）可返回、可重新生成、可修改，不丢状态。
3. 手动路径支持跳过和自下而上使用，空层级不阻断核心闭环。
4. 能量球、滑条、果实、成长树和周结算动效稳定流畅。
5. 弱网下今日清单、本周重点和每日能量仍可使用。
6. 四 tab 结构清晰，不让 AI、统计和复杂任务列表压过主舞台。
7. UI 实现以 `prototype/index.html` 终稿静态原型为参照基准（深色玻璃质感、青绿色能量反馈、琥珀色果实反馈）。

## 2. 工程分层

采用 feature-first 目录结构：

```
app/                        ← expo-router 文件系统路由
  (tabs)/
    energy.tsx              ← 能量页（首页）
    todo.tsx                ← 清单页
    plan.tsx                ← 计划页
    tree.tsx                ← 成长树页
  onboarding/               ← 冷启动流程（全屏，不在 tab 下）
    choose.tsx              ← 路径选择（三路径）
    quick.tsx               ← 快速规划
    vision.tsx              ← 深度愿景
    manual/                 ← 手动 OKR 五层引导
      annual.tsx
      quarter.tsx
      month.tsx
      week.tsx
      today.tsx
  settlement/               ← 周结算流程
  _layout.tsx

src/
  features/
    onboarding/             ← 冷启动规划
      components/
      hooks/
      services/
      types.ts
    energy/                 ← 能量记录
    todo/                   ← 今日清单
    plan/                   ← 计划时间轴
    tree/                   ← 成长树
    settlement/             ← 周结算
    goal/                   ← 目标管理（年/季/月）
  shared/
    components/             ← 通用 UI 组件
    hooks/                  ← 通用 hooks
    utils/                  ← 工具函数
    theme/                  ← 设计 token、颜色、字体
    api/                    ← API 客户端、请求封装
  stores/                   ← Zustand stores
  db/                       ← expo-sqlite 本地数据层
```

每个 feature 内部保持同样分层：`components/`（UI）、`hooks/`（业务逻辑）、`services/`（API 调用）、`types.ts`（类型定义）。

## 3. 导航结构

主 App 固定为 5 个 tab：

1. `能量`：首页，高频入口。
2. `清单`：今日 todolist 和本周 7 天概览。
3. `计划`：月计划按周、年计划按 Q。
4. `成长树`：年度树、果实时间胶囊、荣誉层。
5. `我的`：账号资料、昵称/邮箱展示与编辑、头像入口和退出登录。

特殊流程：

1. 冷启动规划作为首次进入的全屏流程，不挂在底部 tab 下。三条路径各自独立子路由。
2. 周结算作为独立流程，可从能量页、成长树页或演示入口进入。
3. 本周概览可作为清单页内的右侧滑入面板或 modal。
4. 计划页点击周节点跳转到清单页，并直接打开对应周概览。

路由示例：
```
/onboarding/choose              ← 路径选择（三路径）
/onboarding/quick               ← 快速规划
/onboarding/vision              ← 深度愿景
/onboarding/manual/annual       ← 手动：年目标
/onboarding/manual/quarter      ← 手动：季度目标
/onboarding/manual/month        ← 手动：本月目标
/onboarding/manual/week         ← 手动：本周计划
/onboarding/manual/today        ← 手动：今日 ToDo
/(tabs)/energy                  ← 能量页
/(tabs)/todo                    ← 清单页
/(tabs)/plan                    ← 计划页
/(tabs)/tree                    ← 成长树
/settlement/:weekId             ← 周结算
```

## 4. 状态管理

Zustand 负责客户端状态，React Query 负责服务端状态：

| 状态 | 管理方式 | 说明 |
| --- | --- | --- |
| 冷启动流程步骤、输入、AI 草案 | Zustand | 多步骤表单，需要返回不丢状态 |
| 手动路径五层状态和跳过标记 | Zustand | 记录每层是否跳过、是否已填 |
| 当前周、当前季度 | Zustand | 全局时间上下文 |
| 今日能量值、滑条位置 | Zustand | 页面交互态 |
| 今日清单、本周重点 | React Query | 服务端数据 + 本地缓存 |
| 计划时间轴 | React Query | 服务端数据 |
| 成长树、果实 | React Query | 服务端数据 |
| AI 生成状态（loading/error） | React Query mutation | 异步操作 |
| 同步状态 | Zustand | 离线队列和同步进度 |

## 5. 本地数据层

expo-sqlite 负责弱网离线支持，覆盖高频体验数据：

```
本地表：
  local_goals              ← 年/季/月目标
  local_weekly_focuses     ← 本周重点
  local_todos              ← 今日/本周任务
  local_energy_entries     ← 每日能量记录
  local_settlements        ← 周结算快照
  local_tree_data          ← 果实和荣誉层
  local_ai_drafts          ← AI 草案缓存
  sync_queue               ← 待同步队列
```

每条可同步记录保留：
1. `id`：本地 UUID。
2. `remoteId`：服务端 ID，可为空。
3. `createdAt`、`updatedAt`。
4. `deletedAt`：软删除。
5. `syncStatus`：pending、synced、failed。
6. `version`：服务端版本或本地递增版本。

写入策略：
1. 用户操作先写本地，保证界面立即反馈。
2. 同步队列异步推送到后端。
3. AI 生成依赖网络，成功后写入本地草案；用户确认后再转为正式数据。
4. 周结算确认后生成快照，避免历史受后续编辑影响。

### 5.1 端侧 Schema 迁移

expo-sqlite 使用 `PRAGMA user_version` 管理本地 schema 版本：

1. App 启动时检查本地 `user_version`，与代码中的目标版本比较。
2. 若本地版本低于目标版本，按版本号顺序执行迁移脚本。
3. 迁移脚本放在 `src/db/migrations/` 目录，每个版本一个文件（如 `v1.ts`、`v2.ts`）。
4. 迁移只做加列、加表操作，不做删列（保持向前兼容）。
5. 每次迁移在事务中执行，成功后更新 `user_version`。
6. 迁移失败时记录错误日志，不阻断 App 启动，降级为仅网络模式（本地缓存不可用，但核心功能通过 API 仍可使用）。
7. 迁移脚本示例：

```text
src/db/migrations/
  v1.ts    ← 初始表结构
  v2.ts    ← 新增 local_ai_drafts 表
  v3.ts    ← local_todos 加 sourceType 列
  index.ts ← 迁移注册和执行入口
```

## 6. 动画与自绘方案

| 场景 | 技术 | 说明 |
| --- | --- | --- |
| 能量球（发光、脉冲、充电涌入） | react-native-skia | Canvas 自绘，气泡粒子系统 |
| 能量滑条（氮气尾翼、粒子） | reanimated + skia | 滑动手势 + 粒子渲染 |
| 成长树（树干、枝干、树冠） | react-native-skia | 完整自绘，支持点击果实 |
| 果实生成动画 | reanimated | 弹出、成形、亮起的序列动画 |
| 充电百分比跳动 | reanimated | 数字递增动画 |
| 页面转场、弹层 | reanimated | 标准过渡动画 |
| 清单左滑删除 | gesture-handler + reanimated | 手势交互 |
| 周结算仪式感 | reanimated | 逐步展示、柱状图动画 |

动画要求：
1. 能量确认后播放 1-2 秒充电涌入动效。
2. 百分比数字跟随动画递增。
3. 低性能设备允许降低粒子数量。
4. 页面进入时不自动播放强动效，避免打扰。

## 7. 页面模块设计

### 7.1 冷启动规划

冷启动需要支持三条路径：

1. 快速规划：季度目标 -> AI 拆解 -> 本周重点 + 今日清单建议。
2. 深度愿景：五年愿景 -> 年度 OKR -> 4 个季度 OKR -> 首月 4 周承诺 -> 天计划。
3. 手动 OKR：年目标 -> 季度目标 -> 本月目标 -> 本周计划 -> 今日 ToDo（每层可跳过）。

关键交互规则：
1. 返回上一级只负责导航，不清空输入，不触发重新生成。
2. 重新生成只更新当前草案层级。
3. 确认后进入下一层，已确认内容保持稳定。
4. 用户修改过的 AI 草案必须标记为 userEdited，避免后续被无意覆盖。
5. 手动路径的跳过只标记当前层为空，不阻断下游。
6. 手动路径每层可选局部 AI 辅助，只影响当前层及下游。

### 7.2 能量页

能量页只做两件事：展示本周推进状态、记录并修改今日推进度。

组件：
1. `EnergyOrb`：能量球，Skia 自绘，发光脉冲 + 气泡粒子。
2. `WeeklyFocusProgressPanel`：本周重点推进概览，视觉降级（半透明/灰色）。
3. `EnergySlider`：今日推进度滑条，科技感设计，青色发光轨道 + 氮气尾翼粒子。
4. `ConfirmEnergyButton`：确认今日能量，未查看清单时触发温和提醒。

降级逻辑：本周重点为空时（手动路径跳过场景），隐藏重点概览面板，只允许记录主观能量。

### 7.3 清单页

清单页默认展示今日 todolist。

功能边界：
1. 支持新增、编辑、勾选、左滑删除。
2. 右上角进入本周 7 天概览。
3. 顶部轻量展示本周重点摘要。
4. AI 任务只是建议来源，用户始终可改。
5. 本周重点变更后，旧 AI 建议标记失效，但不自动覆盖任务。
6. 即使上层目标全部为空，清单页始终可用。

### 7.4 计划页

计划页是中频视图。

结构：
1. 顶部切换 `月计划 · 按周` 和 `年计划 · 按 Q`。
2. 月计划展示最近 4 周，当前周高亮。
3. 年计划展示 Q1-Q4 的目标、阶段说明和进度。
4. AI 重规划需要用户先输入变化反馈，最多 3 轮迭代。
5. 点击周节点可跳到清单页本周概览。
6. 空层级展示"暂未设置"，并提供补填入口。
7. 根据 source 显示不同动作：AI 来源显示"AI 重规划"，手动来源显示"编辑计划" + "AI 辅助"。

### 7.5 成长树页

成长树是长期反馈视图。

实现：
1. 树主体用 Skia Canvas 自绘。
2. 果实是可点击对象，点击打开时间胶囊。
3. 树阶段由季度决定。
4. 荣誉层由年度累计达标次数决定，只增不减。
5. 复杂统计不要压在树主画面上，通过详情入口展示。

### 7.6 周结算

周结算是独立流程：
1. 开场语。
2. 每日能量回顾。
3. 本周重点推进回顾。
4. 建议周结果。
5. 用户微调最终结果。
6. 周感悟。
7. 果实生成动画。

提交前需要构造结算快照，包含当时的每日能量、重点、任务摘要和用户感悟。

## 8. 错误与空状态

前端错误处理原则：
1. AI 失败时保留用户输入，允许重试，不清空流程。
2. 同步失败时本地数据继续可见，并在状态入口提示。
3. 删除任务后提供短暂撤销机会。
4. 周结算提交失败时保留已填写内容。
5. 网络错误使用温和语气，不使用惩罚式文案。

典型空状态：
1. 今日无任务：允许手动添加，也可触发 AI 建议。
2. 本周无重点：引导去计划页或冷启动结果确认。
3. 成长树无果实：展示当前树苗，不制造失败感。
4. 计划页空层级：展示"暂未设置"和补填入口。
5. AI 生成中：展示轻量生成状态，不伪装聊天机器人。

## 9. 测试策略

前端测试分层：
1. 业务逻辑单元测试：能量均值、季度达标、结算快照生成。
2. Hook 测试：状态管理、数据流。
3. 组件测试：冷启动返回、重新生成入口、清单 CRUD、能量确认提醒。
4. 集成测试：快速规划到能量记录、手动路径跳过到清单使用、周结算到果实生成。

静态原型仍使用 `npx playwright` 做视觉和交互回归；React Native App 进入实现后补充 Jest + React Native Testing Library 测试链路。
