### 2.1 模块内部分层

每个 feature 模块内部采用三层结构：

| 层级 | 目录 | 职责 |
| --- | --- | --- |
| Data | `data/` | API 调用、本地数据库操作、Repository 实现 |
| Domain | `domain/` | 业务状态定义（freezed 类）、业务规则 |
| Presentation | `presentation/` | Screen、Widget、UI 逻辑 |
| Providers | `providers/` | Riverpod Provider 定义，连接 Data 和 Presentation |

依赖方向：`Presentation → Providers → Data`，`Domain` 被所有层引用但不依赖任何层。

## 4. 状态管理设计

### 4.1 Provider 架构

```text
┌─────────────────────────────────────────────┐
│              UI Layer (Widgets)              │
│  读取 Provider 状态，调用 Provider 方法       │
└──────────────────┬──────────────────────────┘
                   │ ref.watch / ref.read
                   ▼
┌─────────────────────────────────────────────┐
│           Provider Layer (Riverpod)          │
│  AsyncNotifier / Notifier / FutureProvider  │
│  管理业务状态，协调 Repository 调用           │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│          Repository Layer                    │
│  封装 API + 本地数据库的读写逻辑             │
│  决定数据来源（本地优先 or 远程优先）         │
└──────────┬───────────────┬──────────────────┘
           │               │
           ▼               ▼
    ┌────────────┐  ┌─────────────┐
    │  Dio (API) │  │ Drift (本地) │
    └────────────┘  └─────────────┘
```

## 5. 网络层设计

### 5.1 Dio 封装

```text
Dio Instance
├── BaseOptions（baseUrl、timeout、headers）
├── AuthInterceptor（自动附加 JWT、处理 401 刷新）
├── LogInterceptor（开发环境日志）
└── ErrorInterceptor（统一错误转换为 ApiException）
```

### 5.2 API 响应格式

与后端约定统一响应结构：

```json
{
  "code": 0,
  "message": "success",
  "data": { ... }
}
```

端侧统一解析，非 0 code 抛出 `ApiException`。

### 5.3 SSE 流式支持

AI 拆解接口使用 SSE 流式响应，端侧通过 Dio 的 `responseType: ResponseType.stream` 接收，逐步更新 UI 展示拆解动画。

### 6.1 Drift 表定义

核心本地表：

```text

├── fruitsJson (TEXT)
├── updatedAt (DATETIME)

```

### 6.2 同步策略

MVP 阶段采用简单的"本地写 + 延迟同步"策略：

1. 用户操作先写入本地 SQLite，UI 立即响应。
2. 后台队列将未同步记录（`syncedAt` 为空或早于 `updatedAt`）批量推送到服务端。
3. 服务端返回确认后更新 `syncedAt`。
4. 冲突处理：以 `updatedAt` 最新的记录为准，MVP 不做复杂合并。

## 7. 动画方案

### 7.1 动画分类

| 动画 | 实现方式 | 复杂度 |
| --- | --- | --- |
| 能量球脉冲发光 | AnimationController + CustomPainter | 中 |
| 充电气泡涌入 | CustomPainter + 粒子系统 | 高 |
| 能量滑条粒子 | Canvas 绘制 | 中 |
| 百分比数字跳动 | Tween + AnimationController | 低 |
| 果实生成动画 | AnimationController + 缩放/透明度 | 中 |
| 成长树渲染 | Rive 或 CustomPainter | 高 |
| 荣誉层特效 | Rive 或 Shader | 高 |
| Tab 切换 | GoRouter 默认过渡 | 低 |
| 清单左滑删除 | Dismissible | 低 |
| 周结算逐步展示 | PageView + AnimationController | 中 |

### 7.2 性能原则

1. 粒子系统使用 `RepaintBoundary` 隔离重绘区域。
2. 成长树如果用 CustomPainter，需要缓存静态部分，只重绘动态元素。
3. 动画帧率目标 60fps，复杂动画降级到 30fps 仍可接受。
4. 成长树考虑使用 Rive 预制动画，减少运行时计算。

## 9. 错误处理

### 9.1 错误分类

| 类型 | 处理方式 |
| --- | --- |
| 网络超时 | Toast 提示 + 自动重试（最多 2 次） |
| 401 未授权 | 尝试刷新 Token，失败则跳转登录 |
| 服务端业务错误 | Toast 展示错误信息 |
| AI 生成失败 | 展示重试按钮 |
| 本地数据库错误 | 静默上报，降级为纯网络模式 |

### 9.2 全局错误边界

使用 `ProviderScope` 的 `observers` 监听 Provider 错误，统一上报。UI 层使用 `AsyncValue` 的 `when` 方法处理 loading / error / data 三态。
