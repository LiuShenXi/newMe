# 技术选型决策

- 日期：2026-04-28
- 产出角色：技术总监
- 决策状态：MVP 阶段采用
- 适用范围：个人成长管理 App 的首个可用版本

## 1. 背景

产品目标是做一款面向个人成长场景的移动 App，核心闭环包括季度目标、本周重点、今日清单、每日能量、周结算、成长树和 AI 拆解。

当前技术目标：

1. Android 和 iOS 都可用。
2. 华为设备能用即可，MVP 阶段不把 HarmonyOS NEXT 原生上架作为硬性目标。
3. 创始人本身是前端，懂一些 NestJS，需要优先考虑开发效率和长期可维护性。
4. 产品核心风险在体验闭环和 AI 拆解质量，不在高并发性能。

## 2. 最终选型

| 层级 | 选型 | 说明 |
| --- | --- | --- |
| 移动端框架 | Flutter | 一套代码覆盖 Android/iOS，动画、自绘和跨端 UI 表现适合能量球、成长树、果实动效。 |
| 移动端语言 | Dart | Flutter 标准语言，声明式 UI 心智与前端开发接近。 |
| 端侧状态管理 | Riverpod | 比 Bloc 更轻，适合 MVP 和中等复杂度业务状态。 |
| 端侧本地数据库 | SQLite + Drift | 适合目标、周重点、任务、能量记录、周结算等结构化离线数据。 |
| 后端框架 | NestJS | 与前端 TypeScript 技术栈接近，模块化清晰，适合 AI 接入和业务快速迭代。 |
| 后端语言 | TypeScript | 降低全栈切换成本，便于复用类型定义和 JSON Schema。 |
| 主数据库 | PostgreSQL | 稳定可靠，适合关系型业务数据、统计查询和长期演进。 |
| ORM | Prisma | TypeScript 体验好，类型安全，MVP 开发效率高。 |
| AI 接入 | 后端统一封装 | 端侧不直连模型，便于控制成本、隐藏密钥、切换模型和管理 prompt 版本。 |
| 部署方式 | Docker + 单体服务 | MVP 优先简单可控，后续再按压力拆分服务。 |

## 3. 前端决策

### 3.1 采用 Flutter

采用 Flutter 作为移动端主框架。

原因：

1. 对 Android/iOS 的支持成熟，能较快产出双端可用 App。
2. 声明式 UI 与前端心智接近，创始人学习成本相对可控。
3. 产品包含能量球充电、果实生成、成长树状态等强视觉反馈，Flutter 的动画和自绘能力匹配度高。
4. 后续如需要 Web 或桌面预览，也有一定扩展空间。

### 3.2 鸿蒙兼容策略

MVP 阶段的鸿蒙目标定义为：华为设备能安装和使用即可。

执行策略：

1. 首期按 Flutter Android 包覆盖华为 Android 兼容设备。
2. 不承诺 HarmonyOS NEXT 原生应用上架。
3. 不在 MVP 主线投入 ArkTS/ArkUI 原生开发。
4. 后续若必须进入 HarmonyOS NEXT 原生生态，再单独立项验证 Flutter OHOS 或 ArkTS/ArkUI 方案。

需要保留的验证项：

1. Flutter Android 包在目标华为设备上的安装、权限、推送、通知和存储表现。
2. 如果未来验证 Flutter OHOS，需要重点检查插件兼容、打包签名、审核稳定性和运行崩溃风险。

### 3.3 不采用 React Native

暂不采用 React Native。

原因：

1. Flutter 在动画、自绘和一致性 UI 上更适合本产品的成长树和能量反馈体验。
2. React Native 对原生桥接和第三方包质量更敏感，长期维护成本不可忽视。
3. 当前产品不是强依赖 Web 代码复用的业务后台类应用。

### 3.4 不采用三端原生

暂不采用 Android 原生 + iOS 原生 + 鸿蒙原生。

原因：

1. MVP 阶段人力成本过高。
2. 产品体验和闭环仍需验证，不适合过早承担三套端侧工程。
3. 鸿蒙 NEXT 原生支持可以作为后续增长阶段的专项，而不是首期前提。

## 4. 后端决策

### 4.1 采用 NestJS

采用 NestJS 作为 MVP 后端框架。

原因：

1. 创始人已有 NestJS 基础，能降低启动成本。
2. TypeScript 对 AI 返回结构、DTO、Schema 校验和接口类型管理更友好。
3. 产品核心需要快速迭代 AI 拆解、计划生成、周结算和数据结构，而不是先追求极限性能。
4. NestJS 的模块化适合把目标、计划、清单、能量、结算、AI 拆解拆成清晰业务模块。

建议模块边界：

1. `AuthModule`：用户身份与登录。
2. `GoalModule`：季度目标、长期方向、目标类型。
3. `PlanModule`：月计划、本周重点、重新规划。
4. `TodoModule`：每日清单、本周清单概览。
5. `EnergyModule`：每日能量记录、本周累计。
6. `SettlementModule`：周结算、季度结算、果实和荣誉层。
7. `AiModule`：模型调用、prompt 模板、结构化输出和重试。

### 4.2 Go 的定位

MVP 阶段不把 Go 作为主后端。

原因：

1. 当前性能压力不是主要瓶颈。
2. AI 编排、结构化 JSON、提示词版本、业务 DTO 迭代在 TypeScript 生态中更顺手。
3. 创始人熟悉 NestJS，会比切换 Go 更快进入产品验证。

Go 可作为后续演进候选，适合这些场景：

1. 高吞吐定时任务。
2. 大量统计计算。
3. 独立推送服务。
4. 成本敏感的后台 worker。
5. 与主业务服务解耦的性能型服务。

## 5. 数据与 AI 架构

### 5.1 数据库

主数据库采用 PostgreSQL。

初期核心实体：

1. 用户
2. 季度目标
3. 目标类型
4. 月计划
5. 本周重点
6. Todo 任务
7. 每日能量记录
8. 周结算结果
9. 果实
10. 季度荣誉层
11. AI 生成记录

### 5.2 本地优先策略

端侧使用 SQLite + Drift 保存核心使用数据。

原则：

1. 今日清单、每日能量记录、本周重点应支持弱网或离线查看。
2. AI 拆解和跨设备同步依赖后端。
3. 本地数据与服务端数据的冲突处理在 MVP 阶段保持简单，以最后修改时间为主要依据。

### 5.3 AI 接入

AI 能力只通过后端调用。

后端负责：

1. 管理模型供应商密钥。
2. 管理 prompt 模板和版本。
3. 将用户目标、本周重点、已有任务转换为模型上下文。
4. 校验模型输出结构。
5. 记录 AI 生成历史，便于排查和优化。

AI 首期只进入两个核心场景：

1. 周初首次拆解。
2. 用户主动重新规划。

不做：

1. 常驻聊天入口。
2. 高频陪聊。
3. 自动替用户判定每日能量分数。

## 6. MVP 架构形态

MVP 采用单体后端架构：

```text
Flutter App
  -> NestJS API
    -> PostgreSQL
    -> AI Provider
```

端侧本地缓存：

```text
Flutter App
  -> SQLite + Drift
```

部署：

```text
Docker
  -> NestJS API
  -> PostgreSQL
```

MVP 阶段暂不强制引入：

1. 微服务
2. Kubernetes
3. Redis
4. 消息队列
5. 独立 Go worker

当出现异步任务、推送、批量结算、AI 生成队列或明显性能压力时，再引入 Redis/队列/worker。

## 7. 关键风险

### 7.1 Flutter 鸿蒙生态风险

风险：Flutter 官方成熟支持 Android/iOS，但 HarmonyOS NEXT 原生生态不是同等确定性。

应对：

1. MVP 不承诺 HarmonyOS NEXT 原生上架。
2. 将鸿蒙目标定义为华为设备可用。
3. 后续用专项 Spike 验证 Flutter OHOS 或 ArkTS/ArkUI。

### 7.2 AI 成本与稳定性风险

风险：AI 生成质量不稳定，调用成本可能随使用增长上升。

应对：

1. AI 只在关键节点触发。
2. 记录 prompt 版本和生成结果。
3. 对模型输出做结构化校验。
4. 对同一上下文增加缓存或复用机制。

### 7.3 本地与云端同步风险

风险：端侧离线修改与服务端数据可能冲突。

应对：

1. MVP 先以单设备体验为主。
2. 关键记录保留 `createdAt`、`updatedAt`、`syncedAt`。
3. 后续跨设备同步成熟后再引入更复杂的冲突合并策略。

### 7.4 架构过度设计风险

风险：过早引入微服务、队列和复杂基础设施会拖慢 MVP。

应对：

1. 首期保持单体。
2. 模块边界清晰，但部署形态简单。
3. 以真实使用压力决定拆分时机。

## 8. 后续演进路线

### 阶段 1：MVP

目标：跑通核心闭环。

技术重点：

1. Flutter 双端基础 App。
2. NestJS 单体 API。
3. PostgreSQL 数据模型。
4. AI 拆解接口。
5. 本地 SQLite 缓存。

### 阶段 2：体验稳定

目标：提升日常使用稳定性和反馈体验。

技术重点：

1. 动画性能优化。
2. 错误上报和日志。
3. AI 生成质量评估。
4. 数据同步可靠性。
5. 推送提醒。

### 阶段 3：平台扩展

目标：根据真实用户和分发需求扩展平台。

技术重点：

1. Flutter OHOS 可行性验证。
2. HarmonyOS NEXT 原生策略评估。
3. Go worker 或队列服务评估。
4. 多端同步和数据导出。

## 9. 当前结论

MVP 阶段采用：

```text
Flutter + Riverpod + SQLite/Drift
NestJS + TypeScript + PostgreSQL + Prisma
后端统一封装 AI 能力
Docker 单体部署
```

这套方案优先服务于：快速验证产品闭环、降低创始人学习成本、保留后续扩展空间。
