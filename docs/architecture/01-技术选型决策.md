# 技术选型决策

- 日期：2026-04-29
- 决策状态：MVP 阶段采用
- 适用范围：个人成长管理 App 首个可用版本

## 1. 背景

产品核心是个人成长闭环，不是重项目管理工具，也不是 AI 聊天助理。技术选型需要优先支持：

1. Android 和 iOS 双端可用，鸿蒙通过卓易通运行 Android 包。
2. 能量球、果实生成、成长树等强视觉反馈。
3. 今日清单、每日能量、本周重点等核心数据弱网可用。
4. AI 在规划和重规划节点稳定输出结构化结果，模型可切换。
5. 创始人为前端/全栈 JS 背景，技术栈需要零语言切换成本。
6. 全 AI 辅助开发，两周出可上架版本，AI 对技术栈的代码生成质量是关键考量。
7. MVP 阶段不过早承担微服务、三端原生、复杂基础设施。

## 2. 最终选型

| 层级 | 选型 | 说明 |
| --- | --- | --- |
| 移动端框架 | React Native + Expo | JS 背景零切换成本，Expo 加速开发体验，AI 生成质量最高 |
| 移动端语言 | TypeScript | 前后端统一，类型共享，AI 编码效率最大化 |
| 状态管理 | Zustand | 比 Redux 轻量，API 简洁，适合 MVP 到中等复杂度 |
| 服务端状态 | @tanstack/react-query | 缓存、重试、乐观更新开箱即用 |
| 本地数据库 | expo-sqlite | 弱网离线支持，结构化查询 |
| 动画引擎 | react-native-reanimated | 60fps UI 线程动画 |
| 自绘引擎 | @shopify/react-native-skia | 能量球、成长树、果实的 Canvas 级自绘 |
| 导航 | expo-router | 文件系统路由，类 Next.js 体验 |
| 后端框架 | NestJS | 模块化清晰，TS 原生，AI 编排友好 |
| 后端语言 | TypeScript | 与前端统一，DTO、Schema、AI 结构化输出类型协作顺手 |
| 主数据库 | PostgreSQL | 稳定可靠，适合关系型业务和长期演进 |
| ORM | Prisma | TypeScript 体验好，MVP 开发效率高 |
| AI 接入 | 后端统一封装，模型无关 | 密钥保护、成本控制、prompt 版本管理、灵活切换供应商 |
| 部署方式 | Docker 单体 + 国内云 | 首期简单可控 |
| 构建方式 | Expo 本地构建 | 不使用 EAS 云端构建，零成本 |

## 3. 前端选型

### 3.1 采用 React Native + Expo

采用 React Native 的主要原因：

1. 创始人是前端/全栈 JS 背景，零语言切换成本。
2. AI 对 TypeScript/React 的代码生成质量远高于 Dart/Flutter，全 AI 开发效率最大化。
3. 一套代码覆盖 Android/iOS，鸿蒙通过卓易通运行 Android 包。
4. react-native-reanimated + Skia 的动画和自绘能力足以实现能量球、成长树和果实动效。
5. Expo 提供 expo-router、expo-sqlite 等开箱即用能力，加速开发。
6. 前后端 TypeScript 统一，类型可以直接共享。

采用 Expo 框架但本地构建：

1. Expo 框架本身免费开源，提供开发便利（文件系统路由、SQLite、Fast Refresh）。
2. 构建走本地 `npx expo run:android` / `npx expo run:ios`，不使用 EAS 云端构建服务。
3. 需要本地安装 Xcode 和 Android Studio，首次配置由 AI 辅助完成。

### 3.2 采用 Zustand + React Query

Zustand 管理客户端状态：
1. 冷启动流程步骤、输入、AI 草案（多步骤表单，需要返回不丢状态）。
2. 当前周、当前季度等全局时间上下文。
3. 今日能量值、滑条位置等页面交互态。
4. 同步状态和离线队列。

React Query 管理服务端状态：
1. 今日清单、本周重点、计划时间轴等服务端数据的缓存和同步。
2. AI 生成等异步操作的 loading/error 状态。
3. 乐观更新和自动重试。

不采用 Redux 的原因是 MVP 业务复杂度不需要 action/reducer 样板。

### 3.3 采用 expo-sqlite

端侧需要支持弱网和低成本记录：

1. 本周重点、今日清单、每日能量的本地读写。
2. 周结算和成长树历史快照的本地查看。
3. 同步水位、脏数据标记、软删除记录。

### 3.4 采用 Skia + Reanimated

能量球、成长树和果实是产品核心视觉体验：

1. @shopify/react-native-skia 提供 Canvas 级自绘能力，适合能量球粒子系统和成长树渲染。
2. react-native-reanimated 提供 60fps UI 线程动画，适合充电动效、数字跳动、页面转场。
3. react-native-gesture-handler 配合 reanimated 实现清单左滑删除等手势交互。

## 4. 后端选型

### 4.1 采用 NestJS

1. 模块边界天然清晰，适合拆分目标、计划、清单、能量、结算、成长树和 AI。
2. TypeScript 对结构化 AI 输出、DTO 校验和接口类型管理友好。
3. 与前端统一 TypeScript，类型可以共享。
4. 单体部署足够支撑 MVP，后续也能按模块拆出 worker 或独立服务。

### 4.2 采用 PostgreSQL + Prisma

1. 数据关系明确：目标、周重点、任务、能量、结算、果实之间存在稳定关联。
2. 周结算、季度结算和年度回看需要可靠聚合。
3. AI 生成记录、prompt 版本和业务快照需要长期留痕。
4. Prisma 能在 MVP 阶段提升建模和迁移效率。

## 5. AI 选型原则

AI 不作为端侧 SDK 接入，统一由后端封装。模型无关设计，支持灵活切换。

后端负责：
1. 管理模型供应商和密钥（Claude、OpenAI、DeepSeek、通义千问等）。
2. 管理 prompt 模板和版本。
3. 控制 AI 调用频率和成本。
4. 校验结构化输出。
5. 记录输入摘要、输出结果、失败原因和重试次数。
6. 切换模型只改配置，不改业务代码。

MVP 的 AI 场景覆盖规划、重规划和手动路径的局部辅助，不做常驻聊天。

## 6. 鸿蒙兼容策略

MVP 对华为设备的目标是"能安装和使用"，不是 HarmonyOS NEXT 原生首发。

执行策略：
1. 首期通过 React Native Android 包覆盖华为 Android 兼容设备（卓易通）。
2. 不在 MVP 主线投入 ArkTS/ArkUI。
3. 不承诺 HarmonyOS NEXT 原生应用上架。
4. 当华为分发成为关键增长点时，再做专项验证。

## 7. 暂不采用方案

| 方案 | 暂不采用原因 |
| --- | --- |
| Flutter | 需要学 Dart，AI 生成质量不如 TS，前后端语言不统一，两周上架风险高 |
| React（Capacitor/Web） | 动画性能天花板低，不适合能量球和成长树的核心体验 |
| Android/iOS 原生 | 人力成本过高，MVP 阶段体验闭环还需验证 |
| 微服务 | 当前没有高并发或团队边界需求 |
| Kubernetes | 单体 Docker 已能覆盖首期部署需求 |
| Redis/消息队列 | 首期 AI 和结算量可同步处理 |
| 端侧直连 AI | 密钥、成本、prompt 版本和输出校验都不可控 |
| EAS 云端构建 | 需要付费，本地构建零成本且功能等价 |

## 8. 当前结论

MVP 技术栈确定为：

```
React Native + Expo + TypeScript + Zustand + React Query
  + react-native-reanimated + @shopify/react-native-skia
  + expo-router + expo-sqlite
NestJS + TypeScript + PostgreSQL + Prisma
后端统一封装 AI 能力（模型无关）
Docker 单体部署 + 国内云服务
Expo 本地构建（不使用 EAS 云端）
```

这套方案优先服务于：全 AI 开发效率最大化、两周出可上架版本、创始人 JS 背景零切换成本、保留后续扩展空间。
