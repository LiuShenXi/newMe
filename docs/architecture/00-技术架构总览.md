## 2. 架构分层

整个系统分为四层，每层职责明确，不越界。

| 层级 | 职责 | 技术栈 | 详细文档 |
| --- | --- | --- | --- |
| 端侧表现层 | UI 渲染、动画、用户交互 | Flutter + Dart | `02-前端架构设计.md` |
| 端侧数据层 | 本地缓存、离线支持、状态管理 | Riverpod + SQLite/Drift | `02-前端架构设计.md` |
| 服务端业务层 | 业务逻辑、API、AI 编排 | NestJS + TypeScript | `03-后端架构设计.md` |
| 数据与 AI 层 | 持久化存储、AI 模型调用 | PostgreSQL + Prisma + AI API | `04-数据与AI架构.md` |

## 3. 端侧架构概览

### 3.1 技术栈

- 框架：Flutter 3.x
- 语言：Dart
- 状态管理：Riverpod
- 本地数据库：SQLite + Drift
- 网络请求：Dio
- 路由：GoRouter

### 3.2 页面结构

App 采用底部 4 Tab 导航，对应产品四层闭环：

```text
确认推进(能量) → 执行任务(清单) → 回顾计划(计划) → 回看成长(成长树)
```

除 Tab 页面外，还有以下独立流程页面：

- 冷启动引导流程（快速规划 / 深度愿景规划）
- 周结算流程
- 季度结算流程

### 3.3 状态管理策略

Riverpod 按业务域划分 Provider：

| Provider 域 | 管理内容 |
| --- | --- |
| goalProvider | 季度目标、年度 OKR、五年愿景 |
| planProvider | 月计划、本周重点 |
| todoProvider | 今日清单、本周清单 |
| energyProvider | 每日能量记录、本周累计 |
| settlementProvider | 周结算、季度结算、果实、荣誉层 |
| treeProvider | 成长树状态、阶段、果实位置 |
| onboardingProvider | 冷启动流程状态 |

### 3.4 离线策略

MVP 以单设备体验为主，离线策略保持简单：

1. 今日清单、每日能量、本周重点写入本地 SQLite，支持弱网查看和操作。
2. AI 拆解、计划生成必须在线，依赖后端。
3. 本地修改在网络恢复后同步到服务端，冲突以 `updatedAt` 最新为准。

详见 `02-前端架构设计.md`。

## 4. 服务端架构概览

### 4.1 技术栈

- 框架：NestJS
- 语言：TypeScript
- ORM：Prisma
- 数据库：PostgreSQL
- 部署：Docker 单体

### 4.2 模块划分

```text
NestJS Application
├── AuthModule        用户身份与登录
├── GoalModule        季度目标、年度 OKR、五年愿景、目标类型识别
├── PlanModule        月计划、本周重点、AI 协作生成与重规划
├── TodoModule        每日清单、本周清单概览、CRUD
├── EnergyModule      每日能量记录、本周累计计算
├── SettlementModule  周结算、季度结算、果实生成、荣誉层累积
└── AiModule          模型调用封装、prompt 模板、结构化输出校验、生成记录
```

### 4.3 API 设计原则

1. RESTful 风格，资源以业务对象命名。
2. 统一响应格式，统一错误码。
3. JWT 认证，无状态。
4. AI 相关接口支持流式响应（SSE），用于拆解动画展示。

详见 `03-后端架构设计.md`。

## 5. 数据架构概览

### 5.1 核心实体关系

```text
User
 ├── Vision (五年愿景)
 ├── AnnualOKR (年度 OKR)
 │    └── QuarterGoal (季度目标)
 │         ├── WeeklyFocus (本周重点)
 │         │    └── Todo (任务)
 │         └── MonthPlan (月计划)
 │              └── WeekPlan (周计划)
 ├── DailyEnergy (每日能量)
 ├── WeekSettlement (周结算)
 │    └── Fruit (果实)
 ├── QuarterSettlement (季度结算)
 │    └── Honor (荣誉层)
 └── AiGenerationLog (AI 生成记录)
```

### 5.2 数据流向

```text
冷启动输入
  → Vision / QuarterGoal
    → AI 拆解
      → WeeklyFocus + Todo
        → DailyEnergy
          → WeekSettlement + Fruit
            → QuarterSettlement + Honor
              → 成长树渲染
```

### 5.3 端云数据分布

| 数据 | 端侧 SQLite | 服务端 PostgreSQL | 说明 |
| --- | --- | --- | --- |
| 今日清单 | ✓ 读写 | ✓ 主存储 | 支持离线操作 |
| 每日能量 | ✓ 读写 | ✓ 主存储 | 支持离线记录 |
| 本周重点 | ✓ 只读缓存 | ✓ 主存储 | 端侧缓存用于展示 |
| 季度目标 | ✓ 只读缓存 | ✓ 主存储 | 端侧缓存用于展示 |
| AI 生成结果 | ✗ | ✓ 主存储 | 必须在线 |
| 周/季结算 | ✗ | ✓ 主存储 | 必须在线 |
| 成长树状态 | ✓ 只读缓存 | ✓ 主存储 | 端侧缓存用于渲染 |

详见 `04-数据与AI架构.md`。

## 6. AI 架构概览

### 6.1 介入场景

AI 只在以下场景介入，不做常驻聊天：

| 场景 | 触发方式 | 输入 | 输出 |
| --- | --- | --- | --- |
| 冷启动深度规划 | 用户选择深度路径 | 五年愿景 | 年度 OKR → 季度 OKR → 4 周承诺 → 天计划 |
| 冷启动快速规划 | 用户选择快速路径 | 季度目标 | 本周重点 + 今日清单建议 |
| 周初拆解 | 用户主动触发 | 季度目标 + 本周重点 + 已有任务 | 本周每日待办建议 |
| 重新规划 | 用户主动触发 | 当前计划 + 用户反馈 | 后续周新计划 |

### 6.2 调用链路

```text
Flutter App
  → POST /api/ai/generate
    → AiModule 组装 prompt（模板 + 用户上下文）
      → AI Provider API（Claude / GPT）
        → 结构化输出校验
          → 存储 AiGenerationLog
            → 返回结果（支持 SSE 流式）
```

### 6.3 目标类型识别

AI 在用户首次设置季度目标时自动识别目标类型，不同类型采用不同拆解策略：

| 目标类型 | 示例 | 拆解策略 |
| --- | --- | --- |
| 结果型 | 看五本书 | 按进度拆解 |
| 项目型 | 做两个 App | 按里程碑拆解 |
| 习惯型 | 早睡早起、晨跑 | 按频次拆解 |

详见 `04-数据与AI架构.md`。

```text

│          云服务器                 │

│  ┌───────────────────────────┐  │

│  │                           │  │
│  │  ┌─────────┐ ┌─────────┐ │  │

│                                 │
│  ┌───────────────────────────┐  │

│  │         :443               │  │
```

### 7.2 MVP 暂不引入

- 微服务拆分
- Kubernetes
- Redis
- 消息队列
- 独立 Go worker

引入时机：出现异步任务积压、推送需求、批量结算性能瓶颈或 AI 生成队列需求时。

详见 `05-部署与运维方案.md`。
