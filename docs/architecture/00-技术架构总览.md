# 技术架构总览

- 日期：2026-04-29
- 适用范围：个人成长管理 App MVP
- 关联文档：`产品需求设计文档.md`、`docs/superpowers/specs/2026-04-29-manual-okr-design.md`

## 1. 架构目标

本产品的技术架构服务于一个核心闭环：

```
冷启动规划（三条路径）
  -> 本周重点
  -> 今日清单
  -> 每日能量记录
  -> 周结算
  -> 每周果实
  -> 季度进化和荣誉层
  -> 年度成长树回看
```

MVP 架构目标不是追求复杂平台能力，而是尽快稳定验证三件事：

1. 用户能否低成本完成规划、执行、记录和结算。
2. AI 拆解能否真实降低规划脑力，而不是制造额外负担。
3. 能量球、果实和成长树能否形成持续正反馈。

## 2. 总体架构

```
React Native App (Expo)
  ├─ UI / Animation / Navigation
  │   ├─ react-native-reanimated (动效)
  │   ├─ @shopify/react-native-skia (能量球/成长树自绘)
  │   └─ expo-router (导航)
  ├─ Zustand (客户端状态)
  ├─ @tanstack/react-query (服务端状态)
  ├─ 业务逻辑层 (hooks + services)
  ├─ 数据层
  │   └─ expo-sqlite (本地离线)
  └─ API 客户端 (类型共享)

React Native App
  ──> NestJS API (模块化单体)
        ├─ Auth / User
        ├─ Goal / Plan / Todo / Energy
        ├─ Settlement / Tree
        ├─ AI Orchestration (模型无关抽象层)
        └─ Sync
            ──> PostgreSQL
            ──> AI Provider (可切换)
```

部署形态：

```
国内云服务器 (阿里云/腾讯云 ECS)
  ├─ Nginx (HTTPS + 反向代理)
  ├─ Docker Compose
  │   ├─ api: NestJS
  │   └─ postgres: PostgreSQL
  └─ 备份脚本

移动端发布
  ├─ 本地 Expo 构建 → Android APK/AAB → 应用市场 + 鸿蒙卓易通
  └─ 本地 Expo 构建 → iOS IPA → TestFlight → App Store
```

MVP 暂不引入微服务、Kubernetes、独立消息队列和独立 worker。模块边界先在代码层清晰，等真实压力出现后再拆部署单元。

## 3. 分层职责

| 层级 | 职责 | MVP 选型 |
| --- | --- | --- |
| 移动端展示层 | 四 tab、冷启动三路径、能量球、滑条、成长树、周结算动效 | React Native + Expo |
| 移动端状态层 | 客户端状态、服务端缓存、异步加载、草案确认、离线数据流 | Zustand + React Query |
| 移动端业务层 | 规划确认、任务 CRUD、能量记录、结算触发等用例 | TypeScript hooks + services |
| 移动端数据层 | 本地缓存、弱网可读写、同步队列 | expo-sqlite |
| 后端 API 层 | 鉴权、DTO 校验、接口编排、错误响应 | NestJS |
| 后端业务层 | 目标、计划、清单、能量、结算、成长树业务规则 | NestJS modules |
| 后端数据层 | 关系型数据、AI 生成记录、同步状态 | PostgreSQL + Prisma |
| AI 编排层 | prompt 版本、模型调用、结构化校验、重试、记录 | NestJS AiModule（模型无关） |
| 运维层 | 构建、部署、备份、日志、监控、回滚 | Docker + GitHub Actions/SSH |

## 4. 核心链路

### 4.1 冷启动规划

三条并列路径：

快速规划链路：
```
输入季度目标
  -> 后端识别目标类型
  -> AI 生成本周重点和今日清单建议
  -> 用户确认或修改
  -> 写入本地和服务端
```

深度愿景链路：
```
五年愿景
  -> 年度 OKR 草案
  -> 4 个季度 OKR 草案
  -> 首月 4 周承诺
  -> 今日清单建议
  -> 用户逐层确认或修改
```

手动 OKR 链路：
```
年目标（可跳过）
  -> 季度目标（可跳过）
  -> 本月目标（可跳过）
  -> 本周计划（可跳过）
  -> 今日 ToDo
  -> 每层可选局部 AI 辅助
```

架构要求：
1. 返回只导航，不清空输入，不自动重算。
2. 重新生成只影响当前 AI 草案层级。
3. 确认只推进下一步，不自动覆盖用户已改内容。
4. AI 结果必须先经过结构化校验，再进入业务表。
5. 手动路径的跳过只标记当前层为空，不阻断下游。
6. 局部 AI 辅助只影响当前层及下游，不反向补上游。

### 4.2 日常执行

```
清单页查看今日任务
  -> 用户手动新增、编辑、勾选、删除
  -> 能量页查看本周重点推进概览
  -> 用户主观记录今日能量
  -> 本周累计实时重算
```

任务完成率不是每日能量分数。任务只提供参考证据，能量值由用户主观确认。

本周重点为空时（手动路径跳过场景），能量页只允许记录主观能量，不展示重点概览。

### 4.3 周结算

```
读取本周每日能量和任务完成情况
  -> 生成建议周结果
  -> 用户确认或微调
  -> 生成本周果实
  -> 写入成长树时间胶囊
```

周结果一旦确认，需要保留结算快照，避免后续任务修改导致历史果实含义漂移。

### 4.4 季度结算

```
读取季度内已确认周结果
  -> 计算季度均值
  -> 年树进入下一阶段
  -> 若均值 >= 80%，新增永久荣誉层
```

树的季度形态由时间驱动；荣誉层由季度达标驱动，只增不减。

## 5. 模块地图

移动端功能模块：

1. `onboarding`：冷启动三路径（深度愿景、快速规划、手动 OKR）、AI 草案确认、手动五层引导。
2. `energy`：能量球、今日能量记录、本周累计、打分前提醒、重点为空降级。
3. `todo`：今日清单、本周 7 天概览、任务 CRUD。
4. `plan`：月计划按周、年计划按 Q、AI 重规划反馈、空层级展示和补填。
5. `tree`：年度成长树、果实时间胶囊、季度荣誉层。
6. `settlement`：周结算流程、最终周结果、周感悟。
7. `goal`：年目标、季度目标、月目标管理。
8. `sync`：本地写入、远端同步、冲突处理。

后端业务模块：

1. `AuthModule`：登录、令牌、用户身份。
2. `UserModule`：用户基础资料和偏好。
3. `GoalModule`：五年愿景、年度 OKR、季度目标、月目标、目标类型。
4. `PlanModule`：月计划、本周重点、重规划版本。
5. `TodoModule`：每日任务、本周清单概览。
6. `EnergyModule`：每日能量记录、本周累计。
7. `SettlementModule`：周结算、季度结算、快照。
8. `TreeModule`：果实、时间胶囊、树阶段、荣誉层。
9. `AiModule`：AI 场景编排、prompt、结构化输出、生成记录、局部辅助。
10. `SyncModule`：端侧同步、水位、冲突处理。

## 6. 数据权责

MVP 采用"端侧可用，云端归档"的策略：

1. 今日清单、本周重点、每日能量在端侧必须可读写。
2. AI 生成、跨设备同步、长期归档和结算汇总依赖后端。
3. 周结算和季度结算结果写入后端后，需要同步回端侧形成历史快照。
4. MVP 先以单设备体验为主，跨设备冲突用 `updatedAt` 和版本号做简单处理。
5. 删除采用软删除字段，避免同步阶段误删历史记录。
6. 手动路径的空层级在端侧和服务端都以 null/空记录表示，不阻断下游查询。

## 7. MVP 非目标

1. 不做常驻 AI 聊天系统。
2. 不做多人协作和组织空间。
3. 不做复杂日历排程和甘特图。
4. 不做微服务拆分。
5. 不做复杂多设备实时合并。
6. 不做大规模 BI 和后台运营平台。

## 8. 演进触发条件

| 触发信号 | 后续动作 |
| --- | --- |
| AI 生成耗时明显影响体验 | 引入任务队列、进度轮询或 SSE |
| 多端同步冲突开始影响真实用户 | 升级冲突合并策略，引入字段级合并或操作日志 |
| 周结算、提醒、批处理增多 | 引入 Redis/BullMQ 或独立 worker |
| RN Skia 动画在中低端设备卡顿 | 粒子降级、性能档位 |
| 华为分发成为增长关键 | 验证 RN 鸿蒙兼容或 ArkTS Spike |
| 后端单体模块耦合变高 | 先做模块内重构，再考虑拆服务 |
