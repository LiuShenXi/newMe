## 3. 模块职责

### 3.1 AuthModule

| 职责 | 说明 |
| --- | --- |
| 用户注册 | 邮箱 + 密码，密码 bcrypt 哈希 |
| 用户登录 | 返回 access_token + refresh_token |
| Token 刷新 | refresh_token 换取新 access_token |
| 当前用户 | @CurrentUser() 装饰器注入用户信息 |

MVP 阶段不做第三方登录（微信、Apple），后续按需扩展。

### 3.2 GoalModule

| 职责 | 说明 |
| --- | --- |
| 五年愿景 CRUD | 每个用户一条愿景记录 |
| 年度 OKR CRUD | 按年份管理 |
| 季度目标 CRUD | 按年份 + 季度管理 |
| 目标类型标记 | AI 识别后写入 type 字段（result/project/habit） |

### 3.3 PlanModule

| 职责 | 说明 |
| --- | --- |
| 月计划管理 | 4 周为一组的计划 |
| 周计划管理 | 每周的具体行动计划 |
| 本周重点 CRUD | 每周 3-5 条重点 |
| 重新规划 | 接收用户反馈，调用 AI 重新生成后续周计划 |

### 3.4 TodoModule

| 职责 | 说明 |
| --- | --- |
| 今日清单 CRUD | 新增、编辑、删除、勾选完成 |
| 本周清单查询 | 按日期范围查询 7 天清单 |
| 批量同步 | 接收端侧离线操作的批量同步请求 |

### 3.5 EnergyModule

| 职责 | 说明 |
| --- | --- |
| 每日能量 upsert | 当天可无限次修改 |
| 本周累计查询 | 返回本周每日能量值和累计均值 |
| 批量同步 | 接收端侧离线记录的批量同步 |

### 3.6 SettlementModule

| 职责 | 说明 |
| --- | --- |
| 周结算数据准备 | 汇总本周每日能量、重点推进、任务完成情况 |
| 周结算确认 | 用户确认最终周结果，生成果实记录 |
| 季度结算 | 计算季度均值，判断是否达标，累加荣誉层 |
| 成长树数据查询 | 返回年度树状态、所有果实、荣誉层 |

### 3.7 AiModule

| 职责 | 说明 |
| --- | --- |
| 模型调用封装 | 统一调用 Claude / GPT，隐藏供应商差异 |
| Prompt 模板管理 | 按场景维护 prompt 模板和版本 |
| 结构化输出校验 | 校验 AI 返回的 JSON 结构是否符合预期 |
| 生成记录 | 记录每次 AI 调用的输入、输出、耗时、模型版本 |
| SSE 流式输出 | 支持流式返回，端侧逐步展示拆解结果 |

## 4. API 设计

### 4.1 统一响应格式

```typescript
// 成功
{
  "code": 0,
  "message": "success",
  "data": { ... }
}

// 分页
{
  "code": 0,
  "message": "success",
  "data": {
    "items": [...],
    "total": 100,
    "page": 1,
    "pageSize": 20
  }
}

// 错误
{
  "code": 40001,
  "message": "季度目标不存在",
  "data": null
}
```

| 范围 | 含义 |

## 5. 认证设计

### 5.1 JWT 双 Token 机制

```text
登录成功
  → 返回 access_token（有效期 2 小时）+ refresh_token（有效期 30 天）

API 请求
  → Header: Authorization: Bearer <access_token>
  → 过期 → 端侧用 refresh_token 换取新 access_token
  → refresh_token 也过期 → 跳转登录页
```

### 5.2 密码安全

- 密码使用 bcrypt 哈希存储，salt rounds = 12。
- 密码最低 8 位，至少包含字母和数字。
- 登录失败不提示"密码错误"或"用户不存在"，统一返回"账号或密码错误"。

## 6. 数据同步设计

### 6.1 同步接口

端侧离线操作通过批量同步接口上传：

```typescript
// POST /api/todos/sync
{
  "operations": [
    { "type": "create", "localId": "xxx", "data": { ... }, "updatedAt": "..." },
    { "type": "update", "serverId": "xxx", "data": { ... }, "updatedAt": "..." },
    { "type": "delete", "serverId": "xxx", "updatedAt": "..." }
  ]
}
```

### 6.2 冲突处理

MVP 采用 Last-Write-Wins 策略：

1. 比较客户端 `updatedAt` 和服务端 `updatedAt`。
2. 较新的覆盖较旧的。
3. 返回最终状态给客户端，客户端更新本地数据。

## 7. 中间件与管道

### 7.1 全局中间件链

```text
请求进入
  → LoggingInterceptor（记录请求日志）
  → JwtAuthGuard（验证 Token，排除白名单路由）
  → ValidationPipe（DTO 校验）
  → Controller → Service → Prisma
  → ResponseTransformInterceptor（统一响应格式）
  → HttpExceptionFilter（统一错误格式）
```

### 7.2 白名单路由

以下路由不需要 JWT 认证：

- POST /api/auth/register
- POST /api/auth/login
- POST /api/auth/refresh

## 8. 日志与监控

### 8.1 日志策略

| 级别 | 内容 |
| --- | --- |
| INFO | 每个请求的方法、路径、状态码、耗时 |
| WARN | 业务异常（如目标不存在、结算重复确认） |
| ERROR | 未捕获异常、AI 调用失败、数据库连接异常 |

MVP 阶段日志输出到 stdout，由 Docker 收集。后续可接入 ELK 或云日志服务。

### 8.2 健康检查

提供 `/api/health` 端点，检查：

1. 应用进程存活
2. PostgreSQL 连接可用
3. 返回应用版本号

## 9. 测试策略

| 层级 | 工具 | 覆盖重点 |
| --- | --- | --- |
| 单元测试 | Jest | Service 层业务逻辑、AI 输出校验 |
| 集成测试 | Jest + Supertest | API 端到端、认证流程 |
| 数据库测试 | Jest + Prisma（测试库） | 复杂查询、结算计算 |

MVP 阶段测试优先级：结算计算逻辑 > AI 输出校验 > API 集成 > 认证流程。
