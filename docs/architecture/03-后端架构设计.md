# 后端架构设计

- 日期：2026-04-29
- 技术栈：NestJS + TypeScript + PostgreSQL + Prisma
- 部署形态：MVP 模块化单体

## 1. 后端架构目标

后端负责四类能力：

1. 可靠保存目标、计划、清单、能量、结算和成长树数据。
2. 封装 AI 生成能力，模型无关，保证 prompt、密钥、模型和结构化校验不暴露给端侧。
3. 支持端侧弱网和后续同步，避免移动端体验完全依赖实时网络。
4. 为周结算、季度结算、长期回看提供稳定业务规则和历史快照。

MVP 采用模块化单体，先保持部署简单，代码内部按业务边界隔离。

## 3. 模块边界

| 模块 | 职责 | 不负责 |
| --- | --- | --- |
| `AuthModule` | 登录、令牌刷新、用户身份上下文 | 业务权限细节 |
| `UsersModule` | 用户资料、偏好、时区 | 目标和计划内容 |
| `GoalsModule` | 五年愿景、年度 OKR、季度目标、月目标、目标类型 | 每日任务 CRUD |
| `PlansModule` | 月计划、本周重点、重规划版本 | AI 模型直接调用 |
| `TodosModule` | 今日任务、本周任务概览、任务 CRUD | 每日能量分数计算 |
| `EnergyModule` | 每日能量记录、本周累计、查看清单提醒状态 | 周结算最终确认 |
| `SettlementsModule` | 周结算、季度结算、结算快照 | 成长树渲染细节 |
| `TreeModule` | 果实、时间胶囊、季度阶段、荣誉层 | 结算均值计算入口 |
| `AiModule` | AI 场景编排、prompt、调用、校验、记录、局部辅助 | 直接落正式业务数据 |
| `SyncModule` | 端侧同步、变更水位、冲突处理 | 业务规则本身 |
| `NotificationModule` | 推送令牌管理、消息下发、推送场景调度 | 推送文案内容（由产品人格定义） |

边界原则：

1. AI 只产出草案，业务模块负责确认和落库。
2. 周结算通过 Settlement 创建快照，再通知 Tree 生成果实。
3. Todo 的完成情况可作为 Energy 的参考数据，但不直接决定能量分数。
4. Plan 中的本周重点是 Todo 和 Energy 的上游锚点。
5. Goals 支持手动 CRUD 年/季/月目标，各层级可独立存在。

## 4. API 形态

### 4.1 版本策略

采用 URL path versioning：

1. 所有接口以 `/api/v1/` 为前缀。
2. MVP 从 v1 开始，仅在出现不兼容变更时升级版本号。
3. 旧版本保留至少 2 个移动端发布周期后下线。
4. 客户端请求头携带 `X-App-Version`（如 `1.0.0`），后端可按版本做兼容判断。

### 4.2 接口形态

MVP 默认采用 REST + JSON。AI 生成若耗时明显，可扩展为任务轮询或 SSE。

核心接口示例（路径省略 `/api/v1` 前缀）：

| 场景 | 方法 | 路径 | 说明 |
| --- | --- | --- | --- |
| 获取当前用户上下文 | GET | `/me` | 用户、时区、当前周信息 |
| 创建/更新愿景 | PUT | `/goals/vision` | 深度规划起点 |
| 设置季度目标 | POST | `/goals/quarters/:quarterId/goals` | 快速规划或手动设置 |
| 设置月目标 | POST | `/goals/months/:monthId/goals` | 手动路径 |
| 生成 AI 草案 | POST | `/ai/generations` | 按场景生成结构化草案 |
| 确认 AI 草案 | POST | `/ai/generations/:id/confirm` | 转为正式业务数据 |
| 局部 AI 辅助 | POST | `/ai/assist` | 手动路径中的局部辅助 |
| 获取本周重点 | GET | `/plans/weeks/:weekId/focuses` | 清单和能量共同使用 |
| 更新本周重点 | PUT | `/plans/weeks/:weekId/focuses` | 标记旧 AI 建议失效 |
| 获取今日任务 | GET | `/todos/today` | 今日清单 |
| 修改任务 | POST/PATCH/DELETE | `/todos` | CRUD 和软删除 |
| 记录今日能量 | PUT | `/energy/days/:date` | 当天可多次修改 |
| 获取本周能量 | GET | `/energy/weeks/:weekId` | 本周累计和日记录 |
| 创建周结算 | POST | `/settlements/weeks/:weekId` | 生成周快照和果实 |
| 获取成长树 | GET | `/tree/years/:year` | 树阶段、果实、荣誉 |
| 同步变更 | POST | `/sync/push` | 端侧脏数据推送 |
| 拉取变更 | GET | `/sync/pull` | 按水位拉取远端变更 |
| 注册推送令牌 | POST | `/notifications/tokens` | 端侧注册 Expo push token |
| 更新推送偏好 | PUT | `/notifications/preferences` | 用户开关推送场景 |

## 5. AI 编排设计（模型无关）

```text
AiModule
  ├─ AiService              ← 场景调度入口
  ├─ ProviderAdapter        ← 模型无关抽象接口
  │   ├─ ClaudeAdapter
  │   ├─ OpenAIAdapter
  │   ├─ DeepSeekAdapter
  │   └─ QwenAdapter        ← 通义千问等国内模型
  ├─ PromptRegistry         ← prompt 模板 + 版本管理
  ├─ OutputValidator        ← JSON Schema / Zod 校验
  └─ GenerationRepository   ← 生成记录存储
```

统一生成入口：

```text
POST /ai/generations
```

请求核心字段：

```json
{
  "scenario": "quick_quarter_plan",
  "input": {},
  "contextVersion": "string",
  "regenerateFromId": "optional-generation-id"
}
```

`scenario` 枚举：

1. `quick_quarter_plan` — 快速季度规划
2. `vision_to_annual_okr` — 愿景到年度 OKR
3. `annual_to_quarter_okr` — 年度到季度 OKR
4. `quarter_to_four_week_commitments` — 季度到 4 周承诺
5. `weekly_focus_to_todos` — 本周重点到清单
6. `replan_future_weeks` — 后续周重规划
7. `manual_local_assist` — 手动路径局部辅助

局部辅助入口：

```text
POST /ai/assist
```

```json
{
  "level": "week",
  "context": "这周最想推进什么的用户输入",
  "existingData": {},
  "options": { "availableDays": 5 }
}
```

局部辅助只生成当前层及下游，不反向补上游。切换模型只需要改配置，不改业务代码。

确认接口由业务模块参与事务：

```text
POST /ai/generations/:id/confirm
```

确认时需要：

1. 校验草案仍匹配当前上下文版本。
2. 合并用户编辑后的内容。
3. 写入正式业务表。
4. 标记生成记录为 confirmed。

## 6. 事务与一致性

需要事务保护的场景：

1. 确认冷启动规划结果，同时创建目标、本周重点、今日任务。
2. 更新本周重点，并标记旧 AI 任务建议失效。
3. 创建周结算，同时生成果实和树时间胶囊。
4. 季度结算，同时更新树阶段和荣誉层。
5. 同步端侧批量变更。
6. 手动路径确认，按实际填写层级批量写入。

一致性规则：

1. 周结果确认后不因任务后续编辑自动变化。
2. 季度荣誉只增不减。
3. AI 重新生成只更新当前草案层级。
4. 用户手动编辑优先于 AI 草案。
5. 删除默认软删除，避免同步时丢历史。
6. 手动路径跳过的层级保持 null，不创建空记录。

## 7. 鉴权与权限

### 7.1 认证方案

采用 Access Token + Refresh Token 双令牌模式：

| 令牌 | 格式 | 有效期 | 存储位置 |
| --- | --- | --- | --- |
| Access Token | JWT（携带 userId、timezone） | 15 分钟 | 端侧 expo-secure-store |
| Refresh Token | 不透明随机字符串 | 30 天 | 服务端 `refresh_tokens` 表 + 端侧 expo-secure-store |

Refresh Token 轮换机制：

1. 每次刷新时旧 Refresh Token 立即失效，签发新的 Access Token + Refresh Token。
2. 若检测到已失效的 Refresh Token 被再次使用，视为令牌泄露，吊销该用户所有 Refresh Token。
3. 服务端 `refresh_tokens` 表记录 tokenHash、deviceId、createdAt、expiresAt、revokedAt。

登录方式：

1. MVP 采用手机号 + 验证码（国内用户主流）。
2. 预留邮箱验证码扩展。
3. 验证码有效期 5 分钟，同一手机号每分钟最多发送 1 次。

### 7.2 用户隔离

1. 所有业务表带 `userId`。
2. Controller 只从认证上下文取用户 ID，不信任客户端传入的 `userId`。
3. 查询必须限定当前用户。
4. AI 生成记录归属当前用户。
5. 同步接口按用户和设备维度维护水位。

## 8. 错误处理

统一错误结构：

```json
{
  "code": "AI_OUTPUT_INVALID",
  "message": "生成结果暂时不可用，请稍后重试",
  "requestId": "req_xxx",
  "details": {}
}
```

错误分类：

1. `VALIDATION_ERROR`：请求参数不合法。
2. `AUTH_REQUIRED`：未登录或令牌失效。
3. `RESOURCE_NOT_FOUND`：资源不存在或不属于当前用户。
4. `CONFLICT`：上下文版本冲突。
5. `AI_OUTPUT_INVALID`：AI 输出结构不合格。
6. `AI_PROVIDER_ERROR`：模型供应商失败。
7. `SYNC_CONFLICT`：同步冲突。

前端展示使用温和文案，后端日志保留技术细节。

## 9. 可观测性

首期至少记录：

1. `requestId`
2. 用户 ID 的脱敏标识
3. 接口路径、耗时、状态码
4. AI 场景、prompt 版本、模型、token 估算、重试次数
5. 同步批次大小和失败原因
6. 周结算和季度结算任务结果

不在普通日志中记录完整愿景、目标和感悟正文。

## 10. 测试策略

后端测试分层：

1. 单元测试：目标类型识别、能量均值、季度达标、荣誉层累积。
2. Service 测试：周结算事务、AI 草案确认、本周重点变更、手动路径批量写入。
3. API 测试：鉴权、DTO 校验、错误结构。
4. Repository 测试：Prisma 查询、软删除、用户隔离。
5. 集成测试：快速规划确认、深度规划确认、手动路径跳过确认、局部 AI 辅助、端侧同步。

AI 测试需要使用固定 mock 输出，避免模型不稳定导致测试飘动。

## 11. 前后端类型共享

采用 monorepo（pnpm workspace），共享包 `packages/shared`：

```text
packages/
  shared/
    src/
      dto/              ← API 请求/响应类型
      enums/            ← 目标类型、来源类型、同步状态等枚举
      constants/        ← 计算规则常量（达标线 80% 等）
      validators/       ← 共享校验规则（Zod schema）
```

共享边界：

1. `packages/shared` 包含前后端都需要的 DTO 类型、枚举和常量。
2. Prisma 生成的类型留在后端，不直接暴露给前端。
3. 前端 SQLite 的类型定义手动维护，与 shared 包的业务类型对齐。
4. 后端 DTO 从 shared 包导入类型，Prisma model 到 DTO 的转换在 service 层完成。
5. AI 结构化输出的 Zod schema 放在 shared 包，前后端共用校验逻辑。

## 12. 推送通知

### 12.1 基础设施

MVP 采用 Expo Notifications（封装 APNs + FCM），零原生推送配置成本。

后端 `NotificationModule` 职责：

1. 管理推送令牌（`push_tokens` 表：token、userId、platform、enabled、createdAt）。
2. 管理用户推送偏好（存储在 `users.preferences_json.notifications`）。
3. 按场景调度推送消息，并通过 Expo Push API 返回发送 tickets 给调度器记录日志。

### 12.2 推送场景

| 场景 | 触发条件 | 默认时间 | 文案风格 |
| --- | --- | --- | --- |
| 每日能量提醒 | 当天未记录能量 | 用户本地时间 20:00 | "用 30 秒记录今天的能量，给明天留一点方向感。" |
| 周结算引导 | 周日进入结算窗口 | 周日上午 10:00 | "本周已经走到尾声，结一颗果实放进成长树。" |
| 召回 | 连续 3 天无 Todo/能量活动 | 用户本地时间 20:00 | "回来补一条今天的状态，让计划重新接上生活。" |

推送规则：

1. 推送文案遵循产品人格（知性温柔大姐姐语气）。
2. 用户可在偏好中按场景关闭推送。
3. 每天最多 1 条推送，避免打扰。
4. 推送时间按用户时区计算。

2026-04-30 实现补充：`NotificationsService` 当前以“单次调度每个 token 最多返回一条消息”落实低打扰策略，优先级为周结算、召回、每日能量。MVP 尚未新增独立发送日志表；`sendDueNotifications()` 返回 Expo tickets，由后续 cron/worker 或运维脚本记录发送结果。

## 13. 周结算触发机制

1. MVP 采用用户手动进入：能量页和成长树页提供"本周结算"入口按钮。
2. 周日晚推送提醒引导用户进入周结算（见推送场景）。
3. 不自动弹出，不强制结算。
4. 未结算的周在下一周仍可补结算（最多补前一周）。
5. 超过补结算窗口的周标记为"未结算"，不生成果实。

## 14. Deep Linking

采用 expo-router 的 deep linking 支持：

1. 推送通知携带路由参数，点击后跳转到对应页面。
2. MVP 路由映射：
   - 每日能量提醒 → `/(tabs)/energy`
   - 周结算引导 → `/settlement/current-week`
   - 召回 → `/(tabs)/energy`
3. MVP 只支持 App 内路由跳转，不做外部 URL scheme。
4. 后续可扩展 Universal Links / App Links 支持分享和外部跳转。
