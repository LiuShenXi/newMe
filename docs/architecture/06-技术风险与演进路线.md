# 技术风险与演进路线

- 日期：2026-04-29
- 产出角色：技术总监
- 适用范围：个人成长管理 App 全生命周期
- 前置文档：`00-技术架构总览.md`、`01-技术选型决策.md`

## 1. 技术风险评估

### 1.1 风险矩阵

| 编号 | 风险 | 影响 | 概率 | 等级 | 阶段 |
| --- | --- | --- | --- | --- | --- |
| R1 | Flutter 鸿蒙生态不确定 | 中 | 中 | 中 | 扩展期 |
| R2 | AI 生成质量不稳定 | 高 | 高 | 高 | MVP |
| R3 | AI 调用成本上升 | 中 | 中 | 中 | 增长期 |
| R4 | 端云数据同步冲突 | 中 | 低 | 低 | MVP |
| R5 | 架构过度设计拖慢 MVP | 高 | 中 | 中 | MVP |
| R6 | 动画性能不达标 | 中 | 中 | 中 | MVP |
| R7 | 单体服务性能瓶颈 | 中 | 低 | 低 | 增长期 |
| R8 | 创始人 Flutter 学习曲线 | 中 | 中 | 中 | MVP |

### 1.2 各风险详细分析与应对

#### R1：Flutter 鸿蒙生态不确定

**风险描述：** Flutter 官方成熟支持 Android/iOS，但 HarmonyOS NEXT 纯血鸿蒙不在 Flutter 官方支持范围内。社区 Flutter OHOS 方案的插件兼容性、打包签名、审核稳定性存在不确定性。

**影响范围：** 华为 HarmonyOS NEXT 设备用户无法使用。

**应对策略：**

1. MVP 阶段不承诺 HarmonyOS NEXT 原生上架，目标定义为"华为设备可用"（通过 Android 兼容层）。
2. 不在 MVP 主线投入 ArkTS/ArkUI 原生开发。
3. 扩展期安排专项 Spike（1-2 周），验证 Flutter OHOS 方案的可行性。
4. 如 Flutter OHOS 不可行，评估 ArkTS/ArkUI 原生开发的成本和收益。

**验证节点：** 扩展期启动前完成 Spike。

#### R2：AI 生成质量不稳定

**风险描述：** AI 模型对"个人成长目标拆解"这类开放性任务的输出质量波动大。可能出现拆解过于笼统、不符合用户实际情况、结构不完整等问题。

**影响范围：** 直接影响产品核心价值——"AI 帮你减轻规划负担"。

**应对策略：**

1. 结构化输出校验：每个场景定义 JSON Schema，不合格自动重试。
2. Prompt 版本管理：模板以代码文件维护，通过 Git 追踪变更。
3. 生成记录：记录每次 AI 调用的输入、输出、耗时、模型版本，便于分析和优化。
4. 目标类型识别：区分结果型/项目型/习惯型目标，采用不同拆解策略。
5. 用户兜底：AI 输出始终是"建议版"，用户可以修改、重新生成或手动创建。

**验证节点：** 第 3 周 AI 优化迭代后评估生成质量。

#### R3：AI 调用成本上升

**风险描述：** 随着用户增长，AI 调用量线性增加，成本可能成为负担。

**影响范围：** 运营成本。

**应对策略：**

1. AI 只在关键节点触发（冷启动、周初拆解、用户主动重规划），不做高频调用。
2. 每用户每天调用上限 20 次。
3. 相同输入 + 相同 prompt 版本的结果缓存 1 小时。
4. 按场景选择模型：简单场景用轻量模型（Haiku/4o-mini），复杂场景用能力模型（Sonnet/4o）。
5. 记录 token 用量，定期分析成本结构。

**验证节点：** MVP 使用 1 个月后分析实际成本。

#### R4：端云数据同步冲突

**风险描述：** 端侧离线修改与服务端数据可能冲突，尤其在多设备场景下。

**影响范围：** 数据一致性。

**应对策略：**

1. MVP 以单设备体验为主，冲突概率极低。
2. 采用 Last-Write-Wins 策略，以 `updatedAt` 最新为准。
3. 关键记录保留 `createdAt`、`updatedAt`、`syncedAt` 三个时间戳。
4. 后续多设备场景成熟后，再引入 CRDT 或更复杂的合并策略。

**验证节点：** 增长期引入多设备支持前。

#### R5：架构过度设计拖慢 MVP

**风险描述：** 过早引入微服务、队列、复杂基础设施会拖慢 MVP 交付。

**影响范围：** 开发速度和验证节奏。

**应对策略：**

1. 首期保持单体 NestJS + Docker Compose。
2. 模块边界清晰（7 个 NestJS Module），但部署形态简单。
3. 不引入 Redis、消息队列、Kubernetes。
4. 以真实使用压力决定拆分时机，不预设。

**验证节点：** 持续关注，MVP 期间不引入非必要基础设施。

#### R6：动画性能不达标

**风险描述：** 能量球充电、成长树渲染、果实生成等动画在中低端设备上可能卡顿。

**影响范围：** 用户体验核心——每日情绪反馈。

**应对策略：**

1. 粒子系统使用 `RepaintBoundary` 隔离重绘。
2. 成长树考虑 Rive 预制动画，减少运行时计算。
3. 动画帧率目标 60fps，复杂动画降级到 30fps 仍可接受。
4. 在目标设备（中端 Android + iPhone）上实测性能。
5. 必要时降级动画复杂度（减少粒子数、简化特效）。

**验证节点：** 第 2 周体验打磨阶段。

#### R7：单体服务性能瓶颈

**风险描述：** 用户增长后，单体 NestJS 可能出现性能瓶颈。

**影响范围：** API 响应速度。

**应对策略：**

1. MVP 阶段用户量极小，单体足够。
2. NestJS 模块边界清晰，后续可按模块拆分。
3. 出现瓶颈时优先垂直扩容（加 CPU/内存）。
4. AI 调用如果成为瓶颈，优先引入任务队列异步化。

**验证节点：** 用户量超过 1000 时评估。

#### R8：创始人 Flutter 学习曲线

**风险描述：** 创始人是前端背景，Flutter/Dart 有学习成本。

**影响范围：** MVP 开发速度。

**应对策略：**

1. Dart 声明式 UI 与前端心智接近，学习曲线相对可控。
2. Riverpod 比 Bloc 更轻量，降低状态管理学习成本。
3. 优先实现核心流程，动画和视觉效果可以渐进增强。
4. 利用 AI 辅助编码加速 Flutter 开发。

**验证节点：** 第 1 周核心闭环开发完成后评估。

## 2. 演进路线

### 2.1 阶段总览

```text
阶段 1：MVP（第 1-3 周）
  → 跑通核心闭环，创始人自用验证

阶段 2：体验稳定（第 4-8 周）
  → 打磨日常体验，提升稳定性

阶段 3：小范围验证（第 9-16 周）
  → 邀请 10-50 人使用，收集反馈

阶段 4：平台扩展（第 17 周+）
  → 根据验证结果决定扩展方向
```

### 2.2 阶段 1：MVP（第 1-3 周）

**目标：** 跑通核心闭环，创始人自用验证。

**技术重点：**

| 项目 | 说明 |
| --- | --- |
| Flutter 基础 App | 4 Tab 结构 + 冷启动流程 |
| NestJS API | 7 个业务模块基础 CRUD |
| PostgreSQL | 核心数据模型 + Prisma 迁移 |
| AI 拆解 | 快速规划 + 深度规划基础接口 |
| 本地 SQLite | 今日清单 + 每日能量离线支持 |
| Docker 部署 | 单服务器 Docker Compose |

**不做：**

- 充电动效
- 成长树复杂渲染
- 周结算仪式感动画
- 推送提醒
- 多设备同步

**交付标准：** 创始人能完整走通"冷启动 → 今日清单 → 每日打分 → 周结算"流程。

### 2.3 阶段 2：体验稳定（第 4-8 周）

**目标：** 让每日体验有足够的情绪反馈，让用户愿意持续使用。

**技术重点：**

| 项目 | 说明 |
| --- | --- |
| 充电动效 | 能量球气泡涌入 + 百分比跳动 + 颜色渐变 |
| 周结算仪式感 | 逐步回顾 + 果实生成动画 |
| 成长树渲染 | CSS/Rive 实现 + 果实时间胶囊 |
| 错误处理 | 全局错误边界 + 用户友好提示 |
| 日志上报 | 关键错误上报 + AI 调用监控 |
| 数据同步 | 离线操作可靠同步 |

**新增基础设施：** 无。保持 MVP 架构不变。

**交付标准：** 创始人连续使用 2 周后愿意继续使用。

### 2.4 阶段 3：小范围验证（第 9-16 周）

**目标：** 邀请 10-50 人使用，收集真实反馈。

**技术重点：**

| 项目 | 说明 |
| --- | --- |
| AI 优化 | 目标类型识别 + 分场景模型选择 |
| 计划页时间轴 | 4 周滚动计划 + AI 协作生成 |
| 产品人格 | 语气系统落地到所有触点 |
| 推送提醒 | 每日温和提醒（FCM/APNs） |
| 性能优化 | 动画帧率 + 启动速度 + 内存占用 |
| 应用分发 | Android 应用市场 + iOS TestFlight |

**可能新增基础设施：**

- Redis（如果推送需要任务队列）
- 云日志服务（如果日志量增大）

**交付标准：** 50 人中有 30% 以上连续使用 4 周。

### 2.5 阶段 4：平台扩展（第 17 周+）

**目标：** 根据验证结果决定扩展方向。

**可能的技术方向：**

| 方向 | 触发条件 | 技术方案 |
| --- | --- | --- |
| 鸿蒙原生 | 华为设备用户占比 > 20% | Flutter OHOS Spike 或 ArkTS/ArkUI |
| 多设备同步 | 用户反馈多设备需求强烈 | CRDT 或 Operational Transform |
| 性能扩展 | 单机 CPU > 70% 持续 | 垂直扩容 → 水平扩展 |
| AI 队列 | AI 调用排队 > 5 秒 | Redis + Bull 任务队列 |
| Go Worker | 批量结算/统计计算慢 | 独立 Go 服务处理计算密集任务 |
| 社交功能 | 产品验证通过，准备扩展 | 新增社交模块 + 消息系统 |

## 3. 技术债务管理

### 3.1 MVP 阶段允许的技术债务

| 债务 | 原因 | 偿还时机 |
| --- | --- | --- |
| 简单的 Last-Write-Wins 同步 | MVP 单设备为主 | 阶段 3 引入多设备前 |
| 无 Redis 缓存 | MVP 用户量极小 | 出现性能瓶颈时 |
| 日志输出到 stdout | MVP 不需要复杂日志系统 | 阶段 3 用户量增长时 |
| 无自动化 E2E 测试 | MVP 快速迭代优先 | 阶段 2 体验稳定后 |
| AI prompt 硬编码在代码中 | MVP 不需要动态模板 | 阶段 3 需要 A/B 测试时 |
| 无灰度发布 | MVP 用户量极小 | 阶段 3 用户量增长时 |

### 3.2 技术债务红线

以下债务在任何阶段都不允许：

1. 密码明文存储。
2. API 密钥硬编码在代码中。
3. SQL 注入风险。
4. 无数据库备份。
5. 无健康检查。
