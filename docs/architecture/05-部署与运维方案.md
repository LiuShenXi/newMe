# 部署与运维方案

- 日期：2026-04-29
- 适用范围：MVP 后端、数据库、移动端发布

## 1. 部署目标

MVP 部署目标：

1. 简单、可复现、可回滚。
2. 支持 NestJS API、PostgreSQL 和 HTTPS。
3. 支持数据库迁移、备份和日志查看。
4. 不为尚未出现的高并发提前引入复杂平台。
5. 为 AI 密钥、数据库密码和应用密钥提供基础安全边界。

## 2. 环境划分

| 环境 | 用途 | 数据 |
| --- | --- | --- |
| local | 本地开发和调试 | 本地假数据 |
| staging | 发布前验证、移动端测试包联调 | 可重置测试数据 |
| production | 正式使用 | 真实数据，必须备份 |

MVP 早期如果只有创始人自用，可以先将 staging 和 production 部署在同一台机器的不同目录或不同 compose project 中，但环境变量和数据库必须隔离。

## 3. MVP 部署形态

```text
国内云服务器 (阿里云/腾讯云 ECS)
  ├─ Nginx
  │   └─ HTTPS / reverse proxy
  ├─ Docker Compose
  │   ├─ api: NestJS
  │   └─ postgres: PostgreSQL
  └─ backup scripts
```

暂不引入：

1. Kubernetes
2. 多实例负载均衡
3. Redis
4. 消息队列
5. 独立 worker
6. 服务网格

当 AI 调用、推送提醒、结算批处理或同步压力明显增加时，再引入 Redis/BullMQ 或独立 worker。

## 4. Docker Compose 建议

服务：

1. `api` — NestJS 编译产物。只暴露给 Nginx 或 Docker 内网。通过环境变量读取数据库、JWT、AI 配置。
2. `postgres` — 不暴露公网端口。挂载数据卷。配置定期备份。
3. `nginx` — 可直接部署在宿主机，也可容器化。负责 HTTPS、反向代理、请求体大小限制。

2026-04-30 实现补充：仓库已新增 `apps/api/Dockerfile`、`docker-compose.yml` 与 `nginx/default.conf`。Compose 通过 `NGINX_PORT` 暴露 Nginx；当前本地联调默认端口为 `37200`，反向代理 `/api/v1/*` 到 API 容器。API 容器启动时先执行 `prisma migrate deploy`，再启动 `node dist/main`。本地 AI 服务在容器内默认通过 `http://host.docker.internal:4100/v1` 访问，GLM fallback 通过 `AI_FALLBACK_*` 环境变量配置。

2026-04-30 运行态验收补充：API 镜像采用多阶段构建，先构建 `@newme/shared` 的 CommonJS 输出，再构建 NestJS API；runtime 阶段复制 API 运行依赖和 Prisma schema，避免容器内缺少 workspace 包或 Prisma Client。根 `package.json` 已通过 `pnpm.onlyBuiltDependencies` 显式允许 Nest、Prisma 与 bcrypt 的必要构建脚本。Nginx 增加 `location = /api/v1/health` 精确转发，已通过 `docker compose config`、`docker compose up --build -d` 与 `http://localhost:8080/api/v1/health` 验证，返回 `status=ok` 且 `database=connected`。

2026-05-04 本地联调端口补充：为避开本机 `8080` 冲突，`docker-compose.yml` 默认 `NGINX_PORT` 已调整为 `37200`，移动端 Web 默认端口为 `37300`，移动端默认 API base URL 为 `http://127.0.0.1:37200/api/v1`。后续本机验收优先使用 `http://localhost:37200/api/v1/health`；如服务器或本机已有端口规划，可通过 `NGINX_PORT` 和 `EXPO_PUBLIC_API_BASE_URL` 覆盖。

关键环境变量：

| 变量 | 说明 |
| --- | --- |
| `NODE_ENV` | runtime 环境 |
| `PORT` | API 服务端口 |
| `DATABASE_URL` | PostgreSQL 连接 |
| `JWT_SECRET` | 令牌签名密钥 |
| `AI_PROVIDER` | AI 供应商标识 |
| `AI_API_KEY` | 模型 API Key |
| `AI_MODEL` | 默认模型 |
| `APP_PUBLIC_API_URL` | 移动端请求的 API 域名 |
| `SENTRY_DSN` | 可选错误上报 |

`.env` 不入库。生产密钥只保存在服务器或云平台密钥系统。

## 5. 数据库迁移

Prisma 迁移规则：

1. schema 变更必须生成 migration 文件并入库。
2. production 部署前先在 staging 跑迁移。
3. 删除字段分两步：先停止读写，再删除字段。
4. 涉及历史数据修复时写一次性脚本，并记录执行时间。
5. 迁移失败必须阻断部署。

部署流程中的数据库步骤：

```text
拉取代码
  -> 安装依赖
  -> 构建 API
  -> 执行 prisma migrate deploy
  -> 重启 API
  -> 健康检查
```

## 6. CI/CD

MVP 可采用 GitHub Actions + SSH 部署。

后端流水线：

1. 安装依赖。
2. 运行 lint。
3. 运行单元测试。
4. 构建 NestJS。
5. 构建 Docker image。
6. 推送或上传到服务器。
7. 执行数据库迁移。
8. 重启服务。
9. 调用健康检查接口。

移动端构建：

1. 本地 `npx expo run:android` 构建 Android 包。
2. 本地 `npx expo run:ios` 构建 iOS 包。
3. Android 包提交应用市场 + 鸿蒙卓易通。
4. iOS 包提交 TestFlight / App Store。
5. 后续可加 GitHub Actions 自动构建。

推送服务配置：

1. Expo Notifications 需要在 Expo 项目中配置 FCM Server Key（Android）和 APNs 证书（iOS）。
2. FCM Server Key 通过 Firebase Console 获取，配置到 Expo 项目的 `app.json`。
3. APNs 证书通过 Apple Developer Portal 获取，上传到 Expo 推送服务。
4. 推送令牌由端侧 `expo-notifications` SDK 获取后注册到后端。
5. 后端通过 Expo Push API 统一下发推送，不直接对接 APNs/FCM。

原型阶段继续使用 `npx playwright` 对 `prototype/index.html` 做交互和截图验证。

## 7. 健康检查

后端提供：

```text
GET /health
```

返回：

1. API 是否启动。
2. 数据库连接是否正常。
3. 当前版本号。
4. 可选：AI provider 配置是否存在，不直接调用模型。

不要在健康检查中暴露密钥、数据库连接串或内部错误堆栈。

## 8. 日志与监控

MVP 必备日志：

1. HTTP 请求日志：路径、状态码、耗时、requestId。
2. 错误日志：错误码、堆栈、用户脱敏 ID。
3. AI 调用日志：场景、模型、prompt 版本、耗时、token、失败原因。
4. 同步日志：批次大小、冲突数量、失败原因。
5. 结算日志：周结算和季度结算执行结果。

监控优先级：

1. API 5xx 错误率。
2. API P95 延迟。
3. AI 调用失败率和平均耗时。
4. PostgreSQL 磁盘使用。
5. 数据库备份是否成功。
6. 移动端崩溃率。

工具可从简单方案开始：Docker logs、Sentry、服务器磁盘告警和定时备份检查。

## 9. 备份与恢复

PostgreSQL 备份策略：

1. production 每日自动备份。
2. 保留最近 7 天每日备份，最近 4 周每周备份。
3. 备份文件加密或放在受控存储。
4. 每月至少做一次恢复演练。

需要备份的数据：

1. PostgreSQL 数据库。
2. 上传文件（如果后续有头像或导出文件）。
3. 生产 `.env` 不作为普通备份散落保存，应由密钥系统或受控文档管理。

恢复演练步骤：

```text
创建临时数据库
  -> 导入备份
  -> 启动 staging API
  -> 验证登录、清单、能量、结算和成长树数据
```

## 10. 安全基线

1. 全站 HTTPS。
2. PostgreSQL 不暴露公网。
3. AI API Key 只存在服务端。
4. JWT secret 定期轮换预留机制。
5. 请求体大小限制，防止异常大输入。
6. 所有业务查询按 `userId` 隔离。
7. 日志不打印完整个人愿景、目标、感悟正文。
8. 管理脚本只在服务器内网或 SSH 下执行。
9. 生产数据库只允许受控账号访问。

## 11. 发布与回滚

发布前检查：

1. 后端测试通过。
2. 数据库迁移在 staging 通过。
3. 移动端关键路径通过。
4. AI 关键场景 mock 和真实调用都验证过。
5. 备份任务最近一次成功。

回滚策略：

1. API 保留上一版 Docker image。
2. 数据库迁移尽量向前兼容，避免需要立即回滚 schema。
3. 如果迁移不可逆，发布前必须额外备份。
4. 移动端发布采用灰度或测试渠道先验证。

## 12. 演进条件

| 信号 | 演进动作 |
| --- | --- |
| AI 生成超过 10-15 秒且用户等待明显 | 引入后台任务和进度轮询/SSE |
| 周结算、提醒、同步任务增多 | 引入 Redis + BullMQ |
| API 单实例成为瓶颈 | 拆分无状态 API 多实例 |
| 数据库压力上升 | 加索引、慢查询优化、读写拆分评估 |
| 移动端崩溃影响使用 | 引入更完整的崩溃上报和性能监控 |
| 华为渠道成为重点 | 验证 RN 鸿蒙兼容或 ArkTS Spike |
