## 1. 部署架构

### 1.1 MVP 部署形态

单服务器 + Docker Compose，简单可控。

```text
┌──────────────────────────────────────────────┐
│                  云服务器                      │
│                                              │
│  ┌────────────────────────────────────────┐  │
│  │           Docker Compose               │  │
│  │                                        │  │
│  │  ┌──────────────┐  ┌───────────────┐   │  │
│  │  │   nestjs-api  │  │   postgres    │   │  │
│  │  │   Node 20     │  │   16.x       │   │  │
│  │  │   :3000       │  │   :5432      │   │  │
│  │  └──────────────┘  └───────────────┘   │  │
│  └────────────────────────────────────────┘  │
│                                              │
│  ┌────────────────────────────────────────┐  │
│  │         Nginx（宿主机或容器）            │  │
│  │   反向代理 + TLS 终止 + 静态资源        │  │
│  │   :443 → nestjs-api:3000               │  │
│  └────────────────────────────────────────┘  │
│                                              │
│  ┌────────────────────────────────────────┐  │
│  │         数据卷                          │  │
│  │   postgres-data → /var/lib/postgresql   │  │
│  │   backup-data → /backups               │  │
│  └────────────────────────────────────────┘  │
└──────────────────────────────────────────────┘
```

### 1.2 服务器配置建议

MVP 阶段用户量极小（创始人自用 + 少量测试用户），配置从低起步：

| 项目 | 建议 |
| --- | --- |
| CPU | 2 核 |
| 内存 | 4 GB |
| 磁盘 | 40 GB SSD |
| 带宽 | 5 Mbps |
| 系统 | Ubuntu 22.04 LTS |

## 2. Docker 配置

### 2.1 docker-compose.yml 结构

```yaml
services:
  api:
    build: .
    ports:
      - "3000:3000"
    environment:
      - DATABASE_URL=postgresql://user:pass@postgres:5432/newme
      - JWT_SECRET=${JWT_SECRET}
      - AI_API_KEY=${AI_API_KEY}
      - AI_PROVIDER=${AI_PROVIDER}
      - NODE_ENV=production
    depends_on:
      postgres:
        condition: service_healthy
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/api/health"]
      interval: 30s
      timeout: 10s
      retries: 3

  postgres:
    image: postgres:16-alpine
    volumes:
      - postgres-data:/var/lib/postgresql/data
    environment:
      - POSTGRES_USER=${DB_USER}
      - POSTGRES_PASSWORD=${DB_PASSWORD}
      - POSTGRES_DB=newme
    restart: unless-stopped
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${DB_USER}"]
      interval: 10s
      timeout: 5s
      retries: 5

volumes:
  postgres-data:
```

### 2.2 NestJS Dockerfile 结构

```dockerfile
# 构建阶段
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npx prisma generate
RUN npm run build

# 运行阶段
FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./
COPY --from=builder /app/prisma ./prisma
EXPOSE 3000
CMD ["node", "dist/main.js"]
```

### 2.3 环境变量管理

| 变量 | 说明 | 存储位置 |
| --- | --- | --- |
| DATABASE_URL | 数据库连接串 | .env（不入库） |
| JWT_SECRET | JWT 签名密钥 | .env（不入库） |
| JWT_REFRESH_SECRET | Refresh Token 密钥 | .env（不入库） |
| AI_API_KEY | AI 模型 API Key | .env（不入库） |
| AI_PROVIDER | AI 供应商（claude/openai） | .env |
| NODE_ENV | 运行环境 | docker-compose.yml |

`.env.example` 入库，`.env` 不入库。

## 3. Nginx 配置

### 3.1 核心配置

```text
反向代理：443 → 3000
TLS：Let's Encrypt 自动续期（certbot）
SSE 支持：关闭 proxy_buffering（AI 流式接口）
CORS：由 NestJS 处理，Nginx 不重复配置
限流：单 IP 100 req/min（MVP 足够）
```

### 3.2 SSE 特殊处理

AI 接口使用 SSE 流式响应，Nginx 需要：

```text
location /api/ai/ {
    proxy_pass http://api:3000;
    proxy_buffering off;
    proxy_cache off;
    proxy_read_timeout 300s;
}
```

## 4. CI/CD 流程

### 4.1 MVP 阶段流程

MVP 阶段采用简单的 GitHub Actions + SSH 部署：

```text
开发者 push 到 main
  → GitHub Actions 触发
    → 安装依赖 + 类型检查 + 单元测试
      → 构建 Docker 镜像
        → SSH 到服务器
          → docker compose pull && docker compose up -d
            → 运行 prisma migrate deploy
              → 健康检查确认
```

### 4.2 GitHub Actions 阶段

```text
Job 1: test
  - npm ci
  - npx prisma generate
  - npm run lint
  - npm run test

Job 2: deploy (depends on test, only on main)
  - docker build + push to registry
  - SSH deploy to server
  - Run migrations
  - Health check
```

### 4.3 Flutter 构建

Flutter App 构建独立于后端 CI：

```text
开发者 push 到 main (或 release tag)
  → GitHub Actions 触发
    → flutter test
      → flutter build apk --release
      → flutter build ios --release (需要 macOS runner)
        → 产物上传到 GitHub Releases 或分发平台
```

MVP 阶段 iOS 构建可以在本地 Mac 完成，不强制 CI。

## 5. 数据库运维

### 5.1 迁移管理

使用 Prisma Migrate 管理数据库迁移：

```text
开发环境：npx prisma migrate dev（生成 + 应用迁移）
生产环境：npx prisma migrate deploy（只应用已有迁移）
```

迁移文件入库，生产环境部署时自动执行。

### 5.2 备份策略

| 项目 | 策略 |
| --- | --- |
| 备份方式 | pg_dump 全量备份 |
| 备份频率 | 每日凌晨 3:00 |
| 保留策略 | 每日备份保留 7 天，每周备份保留 4 周 |
| 备份存储 | 本地 /backups + 异地对象存储 |
| 备份验证 | 每周一次恢复测试（手动） |

备份脚本通过 cron 定时执行：

```text
0 3 * * * /opt/scripts/backup-postgres.sh
```

### 5.3 数据库监控

MVP 阶段保持简单：

- 连接数监控：通过 `/api/health` 检查连接可用性。
- 慢查询日志：PostgreSQL 开启 `log_min_duration_statement = 1000`（记录超过 1 秒的查询）。
- 磁盘空间：cron 定时检查，低于 20% 时告警。

## 6. 监控与告警

### 6.1 MVP 监控方案

MVP 不引入 Prometheus/Grafana 等重型方案，采用轻量级监控：

| 监控项 | 方式 | 告警 |
| --- | --- | --- |
| 服务存活 | 外部 HTTP 探针（UptimeRobot 等） | 邮件/微信通知 |
| API 健康 | /api/health 端点 | Docker healthcheck 自动重启 |
| 错误日志 | Docker logs + 关键字过滤 | 脚本扫描 + 通知 |
| 磁盘空间 | cron + df 检查 | 低于 20% 通知 |
| 数据库连接 | health 端点内检查 | 随服务存活告警 |

### 6.2 日志管理

```text
NestJS 日志 → stdout → Docker 日志驱动
  → docker logs api --tail 100
  → 后续可接入 ELK 或云日志
```

日志格式：JSON 结构化，包含 timestamp、level、message、requestId。

MVP 阶段日志保留策略：Docker 默认 json-file 驱动，限制单文件 50MB，最多 3 个文件。

## 7. 安全基线

### 7.1 服务器安全

| 项目 | 措施 |
| --- | --- |
| SSH | 禁用密码登录，仅密钥认证 |
| 端口 | 只开放 22（SSH）、80（HTTP 重定向）、443（HTTPS） |
| 防火墙 | ufw 或云安全组 |
| 系统更新 | 启用自动安全更新 |

### 7.2 应用安全

| 项目 | 措施 |
| --- | --- |
| HTTPS | 强制 TLS，HTTP 301 重定向到 HTTPS |
| CORS | 限制允许的 Origin |
| Rate Limiting | 单 IP 100 req/min |
| 输入校验 | class-validator 全局 ValidationPipe |
| SQL 注入 | Prisma 参数化查询 |
| 密码存储 | bcrypt 哈希 |
| 密钥管理 | .env 不入库，生产环境通过环境变量注入 |

### 7.3 数据库安全

| 项目 | 措施 |
| --- | --- |
| 网络 | PostgreSQL 不暴露公网端口，只在 Docker 内网通信 |
| 认证 | 强密码 + 独立数据库用户 |
| 权限 | 应用用户只授予必要的表权限 |

## 9. 发布流程

### 9.1 后端发布

```text
1. 开发分支合并到 main
2. GitHub Actions 自动触发 test + deploy
3. 服务器拉取新镜像并重启
4. Prisma 自动执行待应用的迁移
5. 健康检查通过后发布完成
6. 如有问题，回滚到上一个镜像版本
```

### 9.2 Flutter 发布

```text
1. 本地构建 APK / IPA
2. Android：上传到应用市场或分发平台（蒲公英/fir.im）
3. iOS：TestFlight 分发或 App Store 提审
4. MVP 阶段以内部分发为主，不急于上架
```

### 9.3 回滚策略

| 场景 | 回滚方式 |
| --- | --- |
| 后端代码问题 | docker compose 切换到上一个镜像 tag |
| 数据库迁移问题 | 从备份恢复 + 回退代码 |
| 端侧问题 | 发布新版本修复（无法远程回滚） |

## 10. 后续演进

当出现以下信号时，考虑升级部署方案：

| 信号 | 升级方向 |
| --- | --- |
| 单机 CPU/内存持续高位 | 垂直扩容或引入负载均衡 |
| AI 调用排队明显 | 引入 Redis + 任务队列 |
| 推送需求落地 | 引入消息队列 + 推送服务 |
| 多实例部署需求 | 引入 Kubernetes 或容器编排 |
| 日志量增大 | 接入 ELK 或云日志服务 |
| 监控需求增强 | 引入 Prometheus + Grafana |
