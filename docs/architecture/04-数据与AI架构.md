## 1. 数据库选型

| 用途 | 选型 | 说明 |
| --- | --- | --- |
| 服务端主存储 | PostgreSQL 16.x | 关系型业务数据、统计查询 |
| ORM | Prisma | TypeScript 类型安全、迁移管理 |
| 端侧本地存储 | SQLite + Drift | 离线缓存、弱网支持 |

### 2.2 Prisma Schema

```prisma

  onboardingPath      String?  // "quick" | "deep"

  weeklyFocusId String?

  sortOrder     Int       @default(0)
  source        String    @default("manual") // "manual" | "ai"

  updatedAt     DateTime  @updatedAt

// ─── 能量 ───

model DailyEnergy {
  id        String   @id @default(uuid())
  userId    String
  date      DateTime @db.Date
  value     Int      // 0-100
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  user User @relation(fields: [userId], references: [id])

  @@unique([userId, date])
}

// ─── 结算 ───

model WeekSettlement {
  id              String    @id @default(uuid())
  userId          String
  weekStartDate   DateTime  @db.Date
  suggestedResult Int       // 系统建议值 0-100
  finalResult     Int?      // 用户确认值 0-100
  reflection      String?   // 周感悟
  confirmedAt     DateTime?
  createdAt       DateTime  @default(now())
  updatedAt       DateTime  @updatedAt

  user  User   @relation(fields: [userId], references: [id])
  fruit Fruit?

  @@unique([userId, weekStartDate])
}

model Fruit {
  id                String   @id @default(uuid())
  weekSettlementId  String   @unique
  brightness        Float    // 0.0-1.0，由 finalResult 映射
  createdAt         DateTime @default(now())

  weekSettlement WeekSettlement @relation(fields: [weekSettlementId], references: [id])
}

model QuarterSettlement {
  id          String    @id @default(uuid())
  userId      String
  year        Int
  quarter     Int       // 1-4
  avgResult   Float     // 季度周结果均值
  achieved    Boolean   // avgResult >= 80
  confirmedAt DateTime?
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt

  user  User   @relation(fields: [userId], references: [id])
  honor Honor?

  @@unique([userId, year, quarter])
}

model Honor {
  id                    String   @id @default(uuid())
  quarterSettlementId   String   @unique
  cumulativeCount       Int      // 本年度累计达标次数 1-4
  createdAt             DateTime @default(now())

  quarterSettlement QuarterSettlement @relation(fields: [quarterSettlementId], references: [id])
}

// ─── AI 生成记录 ───

model AiGenerationLog {
  id            String   @id @default(uuid())
  userId        String
  scene         String   // "quick_plan" | "deep_vision" | "quarter_breakdown" | "month_breakdown" | "weekly_breakdown" | "replan"
  modelProvider String   // "claude" | "openai"
  modelId       String   // 具体模型 ID
  promptVersion String   // prompt 模板版本号
  input         Json     // 发送给模型的上下文
  output        Json     // 模型返回的结构化结果
  durationMs    Int      // 调用耗时
  tokenUsage    Json?    // { prompt_tokens, completion_tokens }
  createdAt     DateTime @default(now())

  user User @relation(fields: [userId], references: [id])

  @@index([userId, scene])
}
```

### 2.3 关键索引说明

| 表 | 索引 | 用途 |
| --- | --- | --- |
| Todo | (userId, date) | 按日期查询今日/本周清单 |
| DailyEnergy | (userId, date) UNIQUE | 每人每天一条能量记录 |
| WeekSettlement | (userId, weekStartDate) UNIQUE | 每人每周一条结算 |
| WeeklyFocus | (userId, weekStartDate) | 按周查询重点 |
| AiGenerationLog | (userId, scene) | 按场景查询生成历史 |

## 3. 核心计算规则

### 3.1 本周累计进度

```text
weeklyProgress = average(本周已记录的 DailyEnergy.value)
```

每次用户修改今日能量后实时重算。

### 3.2 建议周结果

```text
suggestedResult = average(本周 7 天中已记录的 DailyEnergy.value)
```

未记录的天数不参与计算（不按 0 处理）。

### 3.3 果实亮度

```text
brightness = finalResult / 100.0
```

连续映射，不使用离散档位。0% → 0.0（灰暗），100% → 1.0（耀眼）。

### 3.4 季度结算

```text
quarterAvg = average(该季度所有已确认 WeekSettlement.finalResult)
achieved = quarterAvg >= 80
```

### 3.5 荣誉层累计

```text
cumulativeCount = count(该年度 QuarterSettlement where achieved = true)
```

只增不减，0-4 对应基础态到年度完成态。

## 4. AI 架构

### 4.1 调用链路

```text
端侧请求
  → AiController 接收请求 + 校验参数
    → AiService 组装上下文
      → PromptTemplateService 选择模板 + 填充变量
        → AI Provider API（Claude / GPT）
          → AiOutputValidator 校验返回结构
            → 存储 AiGenerationLog
              → 返回结果（JSON 或 SSE 流）
```

### 4.2 Prompt 模板管理

每个 AI 场景对应一个 prompt 模板，模板包含：

| 字段 | 说明 |
| --- | --- |
| scene | 场景标识 |
| version | 版本号（语义化，如 v1.0.0） |
| systemPrompt | 系统提示词（角色设定、输出格式要求） |
| userPromptTemplate | 用户提示词模板（含变量占位符） |
| outputSchema | 期望的 JSON 输出结构定义 |

模板以 TypeScript 文件维护在代码中，版本通过 Git 管理。MVP 不引入数据库存储模板。

### 4.3 各场景 Prompt 设计

#### 快速规划（quick-plan）

```text
输入：
  - 用户输入的季度目标文本

输出结构：
{
  "goalType": "result" | "project" | "habit",
  "weeklyFocuses": [
    { "content": "...", "sortOrder": 1 },
    ...  // 3-5 条
  ],
  "todaySuggestions": [
    { "content": "...", "sortOrder": 1 },
    ...  // 3-7 条
  ]
}
```

#### 深度愿景 → 年度 OKR（deep-vision）

```text
输入：
  - 五年愿景文本

输出结构：
{
  "annualOkr": [
    {
      "key": "关键目标描述",
      "results": ["关键结果1", "关键结果2", ...]
    },
    ...  // 3-5 个 OKR
  ]
}
```

#### 年度 OKR → 季度 OKR（quarter-breakdown）

```text
输入：
  - 年度 OKR
  - 当前季度

输出结构：
{
  "quarters": [
    {
      "quarter": 1,
      "goals": [
        { "content": "...", "type": "result" | "project" | "habit" }
      ]
    },
    ...  // 4 个季度
  ]
}
```

#### 季度 OKR → 首月 4 周承诺（month-breakdown）

```text
输入：
  - 当前季度目标列表（含类型）

输出结构：
{
  "weeks": [
    {
      "weekNumber": 1,
      "focusSummary": "本周重点概述",
      "actions": ["具体行动1", "具体行动2", ...]
    },
    ...  // 4 周
  ]
}
```

#### 周初拆解（weekly-breakdown）

```text
输入：
  - 季度目标列表
  - 本周重点列表
  - 已有未完成任务（如有）

输出结构：
{
  "dailyTodos": {
    "monday": [{ "content": "..." }, ...],
    "tuesday": [{ "content": "..." }, ...],
    ...
  }
}
```

#### 重新规划（replan）

```text
输入：
  - 当前计划状态
  - 用户反馈文本

输出结构：
{
  "weeks": [
    {
      "weekNumber": N,
      "focusSummary": "...",
      "actions": ["...", ...]
    },
    ...
  ]
}
```

### 4.4 AI 输出校验

每个场景定义 JSON Schema，AiOutputValidator 在收到模型返回后：

1. 解析 JSON（处理 markdown code block 包裹的情况）。
2. 用 JSON Schema 校验结构完整性。
3. 校验业务约束（如 weeklyFocuses 数量 3-5 条）。
4. 校验失败时自动重试一次（相同 prompt），仍失败则返回错误。

### 4.5 SSE 流式输出

AI 拆解接口支持 SSE，用于端侧展示逐步生成动画：

```text
端侧发送：
  POST /api/ai/quick-plan
  Accept: text/event-stream

服务端返回：
  event: token
  data: {"text": "正在分析你的目标..."}

  event: token
  data: {"text": "识别为项目型目标..."}

  event: result
  data: {"goalType": "project", "weeklyFocuses": [...], ...}

  event: done
  data: {}
```

端侧在收到 `token` 事件时展示打字机效果，收到 `result` 事件时渲染最终结果。

### 4.6 模型选择策略

| 场景 | 推荐模型 | 原因 |
| --- | --- | --- |
| 深度愿景 → 年度 OKR | Claude Sonnet / GPT-4o | 需要较强的理解和规划能力 |
| 季度/月/周拆解 | Claude Haiku / GPT-4o-mini | 结构化拆解，速度优先 |
| 快速规划 | Claude Haiku / GPT-4o-mini | 简单场景，低延迟 |
| 重新规划 | Claude Sonnet / GPT-4o | 需要理解用户反馈并调整 |

MVP 阶段先统一使用一个模型，验证效果后再按场景分流。

### 4.7 成本控制

1. 每个用户每天 AI 调用次数上限：20 次。
2. 相同输入 + 相同 prompt 版本的结果缓存 1 小时。
3. 记录每次调用的 token 用量，便于成本分析。
4. 后续可按用户等级设置不同的调用配额。

## 5. 数据同步架构

### 5.1 同步模型

MVP 采用"本地优先写入 + 后台延迟同步"模型：

```text
用户操作
  → 写入本地 SQLite（立即生效）
  → 标记为待同步（syncedAt = null）
  → 后台同步队列检测到待同步记录
    → 批量推送到服务端
      → 服务端处理并返回确认
        → 更新本地 syncedAt
```

### 5.2 同步触发时机

| 时机 | 说明 |
| --- | --- |
| 网络恢复 | 检测到网络从离线变为在线 |
| App 前台恢复 | 从后台切回前台 |
| 定时轮询 | 每 5 分钟检查一次待同步队列 |
| 用户主动刷新 | 下拉刷新等操作 |

### 5.3 冲突场景

MVP 阶段以单设备为主，冲突概率低。处理策略：

| 冲突类型 | 处理方式 |
| --- | --- |
| 同一记录双端修改 | updatedAt 较新的覆盖较旧的 |
| 本地删除 vs 服务端修改 | 删除优先 |
| 服务端删除 vs 本地修改 | 服务端删除优先，本地丢弃修改 |

### 5.4 数据一致性保障

1. 每条记录携带 `createdAt`、`updatedAt`、`syncedAt` 三个时间戳。
2. 本地记录额外携带 `serverId`，用于关联服务端记录。
3. 同步成功后服务端返回最终状态，本地以服务端返回为准更新。

## 6. 数据备份与恢复

### 6.1 MVP 备份策略

| 项目 | 策略 |
| --- | --- |
| PostgreSQL | Docker volume 挂载 + 每日 pg_dump 定时备份 |
| 备份保留 | 最近 7 天每日备份 + 最近 4 周每周备份 |
| 备份存储 | 本地磁盘 + 异地对象存储（如 S3） |

### 6.2 恢复流程

1. 停止 NestJS 服务。
2. 从备份恢复 PostgreSQL 数据。
3. 重启服务。
4. 端侧下次同步时自动拉取最新数据。
