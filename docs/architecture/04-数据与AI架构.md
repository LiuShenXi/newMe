# 数据与 AI 架构

- 日期：2026-04-29
- 适用范围：MVP 数据模型、同步、AI 编排

## 1. 设计目标

数据与 AI 架构需要同时满足：

1. 支撑季度目标、本周重点、今日清单、每日能量、周结算、成长树的完整闭环。
2. 保留用户修改权，AI 只生成草案，不直接替用户做决定。
3. 支持端侧弱网可用和后续云端同步。
4. 让周果实、季度荣誉和年度回看有稳定历史快照。
5. 能追踪 AI 输出质量、prompt 版本和失败原因。
6. 支持手动路径的自下而上使用，各层级可独立存在。

## 2. 核心实体

服务端核心表：

| 实体 | 说明 |
| --- | --- |
| `users` | 用户身份、时区、偏好 |
| `visions` | 五年愿景，深度规划入口 |
| `annual_objectives` | 年度 OKR 草案确认后的正式数据 |
| `quarters` | 年度内 Q1-Q4 时间边界和阶段 |
| `quarter_goals` | 季度目标，含目标类型 |
| `month_goals` | 月目标，手动路径新增层级 |
| `goal_classifications` | AI 识别的结果型、项目型、习惯型等分类记录 |
| `month_plans` | 每次生成的 4 周滚动计划 |
| `week_plans` | 单周计划节点 |
| `weekly_focuses` | 本周 3-5 条重点 |
| `todos` | 每日任务，支持 AI 建议和用户手动创建 |
| `energy_entries` | 每日能量记录 |
| `weekly_settlements` | 周结算最终结果和快照 |
| `tree_fruits` | 每周果实及时间胶囊 |
| `quarter_honors` | 季度达标产生的永久荣誉层 |
| `ai_generations` | AI 生成记录、prompt 版本、输出、状态 |
| `sync_devices` | 端侧设备和同步水位 |
| `refresh_tokens` | Refresh Token 存储、轮换和吊销 |
| `push_tokens` | 推送令牌、设备、平台 |

## 3. 实体关系

```text
User
  ├─ Vision
  ├─ AnnualObjective
  │   └─ QuarterGoal
  │       └─ MonthGoal
  │           └─ WeekPlan
  │               ├─ WeeklyFocus
  │               └─ Todo
  ├─ EnergyEntry (按日期 + 自然周)
  ├─ WeeklySettlement
  │   └─ TreeFruit
  └─ QuarterHonor
```

关键关系：

1. 一个用户一年可以有一个主年度规划版本。
2. 一个季度包含多个季度目标。
3. 一个月可以有月目标（手动路径），也可以为空。
4. 一个周计划包含 3-5 条本周重点。
5. Todo 归属于具体日期，也可关联来源本周重点。
6. EnergyEntry 归属于具体日期和自然周。
7. WeeklySettlement 记录结算时的快照，不只引用实时数据。
8. TreeFruit 从 WeeklySettlement 派生，但保留独立展示字段。
9. 各层级之间的外键允许为 null，支持手动路径的自下而上使用。

### 3.1 month_goals 与 month_plans 的关系

- `month_goals` 是用户层面的月目标（手动或 AI 辅助创建），回答"这个月最重要的方向"。
- `month_plans` 是执行层面的 4 周滚动计划容器，包含 4 个 `week_plans`。
- 关系：一个 `month_goals` 可以关联一个 `month_plans`，但两者可独立存在。
- 手动路径用户可能只有 month_goals 没有 month_plans（没用 AI 生成 4 周计划）。
- AI 路径会同时生成 month_goals 和 month_plans。
- 外键 `month_plans.monthGoalId` 可为 null。

## 4. 本地数据子集

端侧本地覆盖高频体验：

1. 当前愿景和已确认规划摘要。
2. 当前季度目标和月目标。
3. 最近 4 周计划。
4. 当前周重点。
5. 本周 7 天任务。
6. 每日能量记录。
7. 已确认周结算和果实。
8. 当前年度树阶段和荣誉层。
9. AI 草案缓存和同步队列。

端侧不保存完整 AI prompt、模型响应元数据和调试日志。

## 5. 关键字段规则

所有核心业务表保留：

1. `id`
2. `userId`
3. `createdAt`（UTC）
4. `updatedAt`（UTC）
5. `deletedAt`（软删除，UTC）
6. `source`：`manual`、`ai`、`mixed`、`system`
7. `version`

AI 建议类数据额外保留：

1. `aiGenerationId`
2. `promptVersion`
3. `userEdited`
4. `invalidatedAt`

结算快照类数据额外保留：

1. `snapshotJson`
2. `finalScore`
3. `suggestedScore`
4. `confirmedAt`
5. `reflection`

### 5.1 时区处理

1. 服务端所有时间戳存储为 UTC。
2. 用户注册时记录时区（如 `Asia/Shanghai`），存储在 `users.timezone`。
3. "今天"和"本周"的边界按用户时区计算：后端接口接收客户端传入的本地日期（如 `2026-04-29`），服务端用用户时区做校验。
4. 自然周定义：周一 00:00 到周日 23:59（用户本地时间）。
5. 周结算归属：按结算发起时用户所在时区的自然周。
6. MVP 不处理跨时区旅行场景，用户时区以注册时为准，后续可在偏好中手动修改。

## 6. 同步策略

MVP 同步策略保持简单：

1. 用户操作先写本地。
2. 本地记录进入 `sync_queue`。
3. 网络可用时批量推送变更。
4. 服务端返回远端 ID、版本和更新时间。
5. 端侧按水位拉取远端变更。

### 6.1 冲突解决策略

冲突判定与解决：

1. 客户端推送时携带 `version`，服务端比较版本号。
2. 版本号由服务端自增，客户端同步拉取后更新本地 version。
3. 版本号不一致时：服务端版本号大的赢（服务端是权威源）。
4. 时间戳相同时：同样以服务端版本号为准。
5. 批量推送中部分失败：返回逐条结果（成功/冲突/错误），客户端只重试失败项，成功项正常更新本地状态。
6. 结算数据和已确认 AI 草案不参与冲突合并，以首次确认为准。

冲突原则：

1. 同一设备内本地用户操作优先展示。
2. 不允许 AI 结果自动覆盖用户已编辑内容。
3. 周结算确认后视为历史快照，不参与实时合并。
4. 删除为软删除，服务端保留墓碑记录。
5. MVP 先以单设备为主，多设备复杂冲突作为后续演进。

## 7. 计算规则

### 7.1 本周累计进度

来源：本周已记录的每日能量值。

```text
本周累计 = 已记录每日能量之和 / 已记录天数
```

### 7.2 建议周结果

周结算时由系统给出建议值：

```text
建议周结果 = 本周已记录每日能量平均值
```

用户拥有最终确认权。最终周结果写入 `weekly_settlements.finalScore`。

### 7.3 季度结果

```text
季度结果 = 该季度已确认周结果平均值
```

季度均值 >= 80% 时新增一次季度荣誉。

### 7.4 树阶段

树阶段由时间所属季度决定：

1. Q1：起始阶段。
2. Q2：成长阶段。
3. Q3：丰茂阶段。
4. Q4：年度完成阶段。

树阶段不因季度未达标而停滞。

## 8. AI 场景

MVP AI 场景：

| 场景 | 输入 | 输出 |
| --- | --- | --- |
| 快速季度规划 | 季度目标文本 | 目标类型、本周重点、今日清单建议 |
| 愿景到年度 OKR | 五年愿景 | 年度 OKR 草案 |
| 年度到季度 OKR | 年度 OKR | 4 个季度 OKR 草案 |
| 季度到 4 周承诺 | 当前季度 OKR | 首月 4 周承诺 |
| 本周重点到清单 | 季度目标、本周重点、已有任务 | 7 天任务建议 |
| 后续周重规划 | 当前计划、用户反馈、本周结果 | 后续周计划调整建议 |
| 手动局部辅助 | 当前层级上下文、用户输入 | 当前层及下游建议 |

AI 不做：

1. 常驻聊天。
2. 自动替用户打能量分。
3. 自动覆盖用户任务。
4. 未经确认直接改变正式计划。
5. 反向补齐用户跳过的上游层级。

## 9. AI 编排流程

生成流程：

```text
业务模块构造上下文
  -> AiModule 选择场景和 prompt 版本
  -> ProviderAdapter 调用模型（模型无关）
  -> 解析 JSON
  -> Schema 校验
  -> 业务规则校验
  -> 保存 ai_generations
  -> 返回草案
```

确认流程：

```text
用户编辑草案
  -> 提交确认
  -> 校验上下文版本
  -> 业务模块事务落库
  -> ai_generations 标记 confirmed
```

重新生成流程：

```text
用户点击重新生成
  -> 传入 regenerateFromId
  -> 只生成当前层级草案
  -> 不影响下游已确认数据
```

局部辅助流程（手动路径）：

```text
用户在某层级点击 AI 辅助
  -> 收集当前层上下文（用户输入或已有数据）
  -> AI 只生成当前层及下游建议
  -> 不反向补上游
  -> 用户确认后写入
```

### 9.1 前端等待策略

MVP 阶段 AI 调用采用同步 HTTP 请求：

1. 前端超时设置 30 秒，超时后提示"生成时间较长，请稍后重试"。
2. 后端对 AI Provider 的调用超时设置 25 秒（留 5 秒给网络传输和结构化校验）。
3. 深度愿景的多层拆解是逐层调用（用户确认一层再生成下一层），单次调用耗时可控。
4. 前端 loading 状态展示轻量生成动画，不伪装聊天机器人。
5. 演进条件：当 AI 调用稳定超过 10 秒时，引入后台任务 + 轮询/SSE。

### 9.2 重试与容错策略

| 失败类型 | 重试策略 | 最大重试次数 |
| --- | --- | --- |
| Schema 校验失败 | 立即重试，使用修复 prompt | 1 次 |
| 网络超时 / Provider 错误 | 指数退避（间隔 1s、3s） | 2 次 |
| 频率限制（Provider 429） | 按 Provider 返回的 Retry-After | 1 次 |

熔断机制：

1. 滑动窗口 10 分钟内连续失败 5 次，触发 circuit breaker。
2. 熔断期间（5 分钟）该场景 AI 调用直接返回"AI 暂时不可用，请稍后重试"。
3. 熔断恢复后允许单次探测请求，成功则恢复正常。
4. MVP 不做多 Provider failover，但 ProviderAdapter 接口支持后续扩展。
5. 所有重试和熔断事件记录到 `ai_generations` 表。

### 9.3 调用频率限制

| 维度 | 限制 | 超限响应 |
| --- | --- | --- |
| 同一用户同一场景 | 每分钟最多 3 次 | 429 + "生成太频繁了，休息一下再试" |
| 同一用户全场景 | 每小时最多 30 次 | 429 + "今天已经生成很多次了，休息一下吧" |

实现方式：

1. MVP 用内存计数器（单实例足够），按 userId + scenario 维度计数。
2. 后续可迁移到 Redis 实现分布式限流。
3. 频率限制在 AiService 入口统一拦截，不在各业务模块重复实现。

## 10. 结构化输出

AI 输出必须使用 JSON Schema 或 Zod schema 校验。示例结构：

```json
{
  "goalType": "project",
  "weeklyFocuses": [
    {
      "title": "完成登录模块前后端闭环",
      "reason": "这是本周推进项目可用性的关键结果"
    }
  ],
  "todayTodos": [
    {
      "title": "梳理登录接口字段",
      "estimatedMinutes": 30,
      "sourceFocusTitle": "完成登录模块前后端闭环"
    }
  ]
}
```

校验失败处理：

1. 自动重试一次，使用更严格的修复 prompt。
2. 仍失败则返回可重试错误。
3. 记录失败原因和原始输出摘要。
4. 不把失败输出写入正式业务表。

## 11. Prompt 版本管理

每个 AI 场景需要有独立 prompt 版本：

```text
quick_quarter_plan:v1
vision_to_annual_okr:v1
annual_to_quarter_okr:v1
quarter_to_four_week_commitments:v1
weekly_focus_to_todos:v1
replan_future_weeks:v1
manual_local_assist:v1
```

`ai_generations` 记录：

1. `scenario`
2. `promptVersion`
3. `model`
4. `inputHash`
5. `outputJson`
6. `status`
7. `errorCode`
8. `latencyMs`
9. `tokenUsage`
10. `confirmedAt`

## 12. 成本与隐私

成本控制：

1. 只在用户主动触发规划、重新规划或局部辅助时调用 AI。
2. 对相同上下文可缓存短期结果。
3. 限制每个草案层级的重新生成频率。
4. 记录 token 使用，按场景观察成本。

隐私原则：

1. 端侧不保存模型 API Key。
2. 日志不直接打印完整愿景、目标和感悟正文。
3. AI 生成记录只保存必要排查信息。
4. 后续如支持导出或删除账号，需要能删除用户个人内容和 AI 记录。
