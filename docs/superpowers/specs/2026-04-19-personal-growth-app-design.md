# 个人成长管理 App 设计稿

- 日期：2026-04-19
- 最后更新：2026-04-22
- 状态：原型完成，准备进入开发
- 产出类型：产品设计规格文档

## 1. 项目概述

这是一款面向个人成长场景的移动 App。它不把自己定义成一个功能繁重的效率工具，也不把自己做成一个 AI 聊天助理，而是一个以低脑力规划、每日主观推进确认、长期成长可视化反馈为核心的个人管理产品。

产品主目标不是让用户管理更多任务，而是让用户更轻松地回答三件事：

1. 我这个季度最重要的目标是什么？
2. 我这周最重要的承诺是什么？
3. 我今天到底把这周推进了多少？

产品的长期叙事是一棵年度成长树。用户每天注入能量、每周结算果实、每季度推动树进入下一阶段，并在季度达标时为树叠加永久荣誉特效。

## 2. 问题定义

目标用户有长期成长目标，执行力并不差，但不想被复杂规划耗尽精力。现有产品通常有三个问题：

1. 规划过程复杂，操作成本高，做计划本身就消耗大量脑力。
2. 产品过于任务化，缺少“这一周整体推进感”的主舞台。
3. 缺少可感知、可积累、可长期回看的正反馈。

因此，本产品要解决的核心问题是：

> 如何在不增加思考负担的前提下，帮助有长期成长目标的人，以极低成本完成每周规划与每日推进，并把长期成长设计成可见、可积累、可奖励的反馈系统。

## 3. 目标用户

### 核心用户

- 有长期成长目标，但讨厌复杂效率系统的人
- 工作或生活本身已经很耗脑，不想再把精力花在“怎么规划”上的人
- 需要正反馈来强化坚持，而不是只看待办清单的人

### 典型画像

- 独立开发者、自我驱动学习者、健身/阅读/副业长期推进者
- 愿意做周规划，但不喜欢被细碎任务和复杂视图绑住
- 可以接受主观自评，但希望自评有参照物，而不是纯空想

### 非目标用户

- 重度项目管理用户
- 强依赖复杂日历、甘特图、多人协作流程的用户
- 希望产品变成高频 AI 聊天陪练的人

## 4. 产品原则

1. 简单优先于全面：主界面只承载最核心动作，不承担所有信息。
2. 主观推进感优先于任务数：用户每天记录的是“我推进了多少”，不是“我做了几条任务”。
3. 周承诺是锚点：主观记录必须锚定到本周承诺，不能成为无依据打分。
4. AI 在幕后，不在前台：AI 负责拆解和重规划，不负责聊天陪伴。
5. 正反馈是核心，不是装饰：能量球、果实、成长树是产品闭环的重要部分。
6. 历史成绩不回退：季度达标获得的荣誉效果永久保留，不因后续季度失利而消失。

## 5. 非目标与边界

本产品在 MVP 阶段明确不做以下内容：

- 不做常驻 AI 聊天入口
- 不做首页长任务列表
- 不做复杂月历/周历排程
- 不做多人协作
- 不做自动替用户判定每日完成度
- 不做重度游戏化数值系统

## 6. 产品结构

底部导航固定为 4 个 tab：

1. `能量`
   - 每日高频入口
   - 展示本周能量球（带充电涌入动效）
   - 记录并修改今日推进度
2. `清单`
   - 承载本周承诺（作为任务容器）
   - 展示 AI 生成的任务建议
   - 支持用户手动 CRUD
3. `计划`
   - 纵向时间轴视图，每周一个节点
   - 展示当前季度乃至年度的计划全貌
   - 用户与 AI 协作生成，每次生成 4 周（一个月）的计划
   - 支持用户回顾和调整
4. `成长树`
   - 展示年度树（低频查看场景）
   - 查看每周果实、季度达标结果、年度累计成长记录

这个结构对应产品的四层闭环：

`确认推进 -> 执行任务 -> 回顾计划 -> 回看成长`

## 7. 核心对象模型

### 7.1 季度目标

- 时间范围：Q1 / Q2 / Q3 / Q4
- 作用：定义该季度最重要的目标集合
- 用途：后续生成本周承诺和 AI 拆解任务的输入源
- 目标类型识别：AI 需要识别目标类型并采用不同的拆解逻辑
  - **结果型目标**（如"看五本书"）→ 按进度拆解（第几章、第几本）
  - **项目型目标**（如"做两个 APP"）→ 按里程碑拆解（功能模块、开发阶段）
  - **习惯型目标**（如"早睡早起、晨跑"）→ 按频次拆解（每周 X 次）
- 类型识别时机：用户首次设置季度目标时，AI 自动识别并标记类型

### 7.2 本周承诺

- 数量：每周 3-5 条
- 作用：本周最重要的结果锚点
- 来源：由用户从季度目标中提取
- 特点：高于普通任务，不是任意待办

### 7.3 Todo 任务

- 来源：AI 基于本周承诺生成建议版任务
- 结构：按承诺分组
- 用户能力：新增、删除、编辑、调整顺序、勾选完成
- 定位：执行层，不直接决定每日能量百分比

### 7.4 每日能量记录

- 范围：0% - 100%
- 性质：用户主观评估
- 依据：参考当天任务完成情况和本周承诺推进情况
- 修改规则：当天可无限次修改并实时重算当前周累计值
- MVP 约束：不提供跨周历史回填
- 打分前提醒：用户打分前系统检查是否查看过今日 TDL，若未查看则温和提醒（非强制阻断）
- 推荐打分：不提供系统推荐分数，因为每项任务复杂度差异大且用户可能临时调整计划，必须由用户自主评估

### 7.5 本周结果

- 作用：生成每周果实
- 结算方式：周末统一确认
- 推荐规则：
  - 系统根据本周已记录的每日能量值，给出一个建议周结果
  - 用户可确认或手动调整最终周结果
  - 最终周结果一旦确认，生成本周果实

### 7.6 年度成长树

- 一年只对应一棵树
- 树会按季度进入四个连续成长阶段
- 每周结果在树上长出一颗果实
- 季度达标为树叠加永久荣誉效果

## 8. 端到端核心流程

## 8.1 首次进入与季度设置

1. 用户首次进入 App
2. 创建当前季度目标
3. 可选填写长期方向信息，例如 5 年愿景
4. 系统将当前季度目标设为本阶段唯一规划源

MVP 规定：

- 5 年愿景是可选项，不阻塞主流程
- 季度目标是必填项

## 8.2 周初生成本周承诺

1. 用户进入 `清单` 页
2. 从当前季度目标中整理出本周 3-5 条承诺
3. 这些承诺成为本周唯一锚点

规则：

- 本周承诺不是普通待办
- 本周承诺变更后，原 AI 任务建议需要失效并重算

## 8.3 AI 拆解与清单生成

1. 用户点击 `AI 帮我拆解`
2. AI 读取：
   - 当前季度目标
   - 本周承诺
   - 已有未完成任务（若存在）
3. AI 输出一版按承诺分组的任务建议
4. 用户在 `清单` 页进行手动 CRUD

AI 只在两种场景介入：

1. 周初首次拆解
2. 用户主动点 `重新规划`

AI 明确不做：

- 日常陪聊
- 首页常驻问答
- 每日主动打扰
- 自动替用户判分

## 8.4 日常执行与能量记录

1. 用户在 `清单` 页查看今天的执行任务
2. 完成任务后，回到 `能量` 页
3. 依据当天对本周承诺的推进情况，主观拉动今日能量值
4. 系统实时更新本周累计进度
5. 当天内允许无限次修改

产品规则：

- 每日能量值是主观推进感，不等同于任务完成率
- 任务是参考证据，能量是用户对本周推进的最终判断

## 8.5 周结算

1. 到周末进入周结算流程
2. 系统以仪式感的方式逐步回顾本周：
   - 一步步展示每天的推进情况（"周一你做了这些…周三你推进了那个…"）
   - 展示本周承诺完成概览
   - 展示每日能量记录曲线
   - 给出建议周结果
3. 用户确认或微调最终周结果
4. 果实生成动画：从树枝上冒出来、慢慢成形、最终亮起来
5. 结算完成后，展示安静画面：这周的树，带着新长出的果实

周结算语气示例："这一周辛苦了，我们一起看看这周的收获吧。"

果实规则：

- 每周仅生成 1 颗果实
- 果实亮度与最终周结果连续对应
- 0% 接近灰暗
- 100% 接近耀眼发亮
- 中间值按连续亮度变化，不使用离散档位

## 8.6 季度结算

1. 季度结束时，计算该季度所有周结果均值
2. 年树无论是否达标，都进入下一个季度形态
3. 若季度均值 `>= 80%`，则树额外获得一层永久荣誉效果

季度规则：

- 形态进化由时间驱动
- 荣誉累积由季度达标驱动
- 季度荣誉不会因为后续季度失利而消失

## 8.7 年度回看

用户可以在 `成长树` 页回看：

- 整棵树一年四个阶段的变化
- 每一周留下的果实
- 年度内累计达标次数
- 荣誉层累积情况

## 9. 清单页设计规则

清单页不是普通 todo 页，而是”把季度目标压缩成每周承诺，再由 AI 变成执行动作”的地方。

信息结构采用”承诺作为容器”的层级设计：

1. `本周承诺区`
   - 3-5 条
   - 固定置顶
   - 每条承诺是一个可展开的容器
   - 表示本周最重要的结果
2. `承诺内的任务列表`
   - 点击展开某条承诺，才看到该承诺下的任务
   - 首次由 AI 生成建议任务
   - 支持用户 CRUD
   - 支持勾选完成

通过交互层级传达”承诺高于任务”的关系，用户无需理解概念定义即可自然感知两者区别。

清单页原则：

- 承诺是容器，任务是容器内的执行项
- 任务服务于承诺
- 承诺变更后，任务建议需重新生成

## 10. 能量页设计规则

能量页是 App 的首页和每日高频入口。

它只负责两件事：

1. 展示本周推进状态
2. 记录并修改今日推进度

### 10.1 能量球充电动效

用户确认今日能量后，能量球触发充电涌入动效：

- 气泡从底部不断涌入能量球内部（类似手机充电时的电量涌入效果）
- 能量球的百分比数字实时跳动更新
- 能量球颜色随百分比渐变（低值偏冷色/暗色，高值偏暖色/亮色）
- 整个过程持续 1-2 秒，给用户"注入能量"的瞬间反馈感

这个动效是每日体验的情绪高点，确保用户每天都能获得一次微小的正反馈。

### 10.2 本周承诺完成概览（打分参照物）

用户打分前，能量页需要展示"本周承诺完成概览"作为参照物，避免纯凭感觉打分。

页面布局结构：

```
┌─────────────────────────────┐
│   本周能量球（带百分比）      │
│   （充电动效区域）            │
└─────────────────────────────┘

┌─────────────────────────────┐
│   本周承诺完成概览            │
│   ✓ 完成 XX 功能前端 (60%)   │
│   ✓ 读完第 3-5 章 (40%)      │
│   ✓ 晨跑 5 次 (已完成 3/5)   │
└─────────────────────────────┘

┌─────────────────────────────┐
│   今日推进度滑条              │
│   0% ←──────●────→ 100%     │
└─────────────────────────────┘
```

展示规则：
- 每条承诺显示当前完成进度（百分比或次数）
- 进度来源于清单页的任务完成情况和用户手动更新
- 用户看到具体进度后，再对今日推进度做主观评估

### 10.3 页面规则

流程要求：

- 每次能量修改都实时反映到本周累计值
- 首页不出现长任务列表
- 首页不出现 AI 对话框
- 首页不承载复杂统计

## 11. 成长树系统规则

## 11.1 树的时间成长

年度只有一棵树，它按季度自然进入四个阶段：

1. Q1：起始阶段
2. Q2：成长阶段
3. Q3：丰茂阶段
4. Q4：年度完成阶段

季度切换后，树一定进入下一阶段，不设置“未达标则卡住”的逻辑。

## 11.2 每周果实

- 每周 1 颗果实
- 果实的亮度、饱满度和通透感由本周结果决定
- 果实承担”周完成质量”的表达

## 11.3 果实时间胶囊功能

成长树不只是视觉装饰，而是”我这一年做了什么”的时间胶囊。

交互设计：
- 用户点击树上的某颗果实，弹出那一周的回顾卡片
- 回顾卡片内容：
  - 该周日期（如”第 12 周（2026-03-15）”）
  - 本周承诺列表
  - 本周结果百分比
  - 本周感悟（用户在周结算时可选填写）

示例卡片：

```text
┌─────────────────────────────┐
│   第 12 周（2026-03-15）     │
│                              │
│   本周承诺：                  │
│   • 完成登录模块前后端        │
│   • 读完《人类简史》第 5-7 章 │
│   • 晨跑 5 次                 │
│                              │
│   本周结果：75%              │
│   本周感悟：（用户当时写的）  │
└─────────────────────────────┘
```

这个功能让用户在任何时候都能回顾”我那一周在推进什么”，强化长期成长的可见性。

## 11.4 季度荣誉层

季度达标标准：

- 季度所有周结果的均值 `>= 80%`

当季度达标时，不改变树是否进化，而是在进化后的树上叠加永久荣誉层。

荣誉层按“本年度累计达标次数”累加：

1. `0 次达标`
   - 基础树态
2. `1 次达标`
   - 发光
3. `2 次达标`
   - 流光溢彩
4. `3 次达标`
   - 更强一层奇幻荣誉效果
5. `4 次达标`
   - 年度完成态

荣誉层原则：

- 只增不减
- 表示年度内累计赢过几次
- 是永久荣誉，不是短期状态

## 12. 计划页设计规则

计划页是用户回顾和管理季度/年度计划的入口，定位为中频使用场景。

### 12.1 视图形态

- 纵向时间轴，每周一个节点
- 每个节点旁展示该周的计划承诺和完成状态
- 当前周高亮标记，已过去的周显示完成情况，未来的周显示计划内容
- 可上下滚动查看整个季度乃至年度的计划全貌

### 12.2 AI 协作生成计划

- 用户与 AI 协作讨论生成计划（AI 出一版 → 用户提意见 → AI 修改，最多 3 轮迭代）
- 每次只生成 4 周（一个月）的计划，避免过远的计划因变动而失效
- 当用户未完成某周计划或计划有变时，可触发 AI 重新规划后续周
- 计划页的本周承诺与清单页的本周承诺保持同步

### 12.3 计划页与清单页的关系

- 计划页展示宏观时间线和每周承诺概览
- 点击某周节点可跳转到清单页查看该周的详细任务
- 本周承诺在两个页面保持数据一致

## 13. 冷启动体验设计

### 13.1 设计原则

先让用户"感受到产品是什么"，再让他"认真使用产品"。不在第一次打开时要求用户做深度思考。

### 13.2 冷启动流程

第一步：用户打开 App，看到一棵小树苗的画面，配一句话——"这是你今年的成长树，它会陪你走过四个季节。"

第二步：问一个简单的问题——"这个季度，你最想推进的一件事是什么？"一个输入框，一个问题，零压力。不出现"请设置季度目标"这种表单感的引导。

第三步：用户写完，立刻触发 AI 拆解，生成本周承诺 + 任务建议。用户马上体验到核心闭环——"原来这个 App 是帮我把目标变成每周行动的"。

第四步：引导用户记录第一次能量——"今天在这件事上，你推进了多少？"配合充电动效，让用户第一次就体验到每日反馈循环。

第五步：在用户使用 2-3 天后，温和引导——"你还有其他想推进的方向吗？可以补充到季度目标里。"

### 13.3 核心思路

先用一件事跑通完整闭环（目标 → 承诺 → 任务 → 能量 → 反馈），再逐步扩展到完整的季度规划。降低首次使用门槛，提高首次体验完整度。

## 14. 产品人格

### 14.1 人格定位

知性温柔的大姐姐——不催促、不评判、不居高临下，但始终在。

### 14.2 各触点的语气表现

- 每日提醒（推送）：不是"该记录了！"，而是"今天过得怎么样？有空的话记录一下吧。"
- 打分前 TDL 提醒：不是"你还没完成清单"，而是"看起来今天的清单还没看过，要不要先看看再打分？"
- 周结算引导语：不是"请确认本周结果"，而是"这一周辛苦了，我们一起看看这周的收获吧。"
- 连续几天未打开时的召回：不是"你已经 3 天没记录了"，而是"好几天没见了，最近还顺利吗？"
- 季度达标时：不是"恭喜达标！"，而是"这个季度你真的很棒，树记住了你的努力。"
- 冷启动引导：不是"请设置你的季度目标"，而是"这个季度，你最想推进的一件事是什么？"

### 14.3 人格原则

1. 温和但不敷衍——每句话都有实际意义，不是空洞的鼓励
2. 陪伴但不打扰——只在关键节点出现，不做日常高频推送
3. 理解但不替代——理解用户的处境，但不替用户做决定
4. 记住但不翻旧账——记住用户的努力，不提及用户的失败

## 15. 数据与计算规则

### 15.1 本周累计进度

- 来源：用户每日能量记录
- 更新方式：每次修改今日能量后实时更新

### 15.2 建议周结果

- 默认来源：本周已记录的每日能量值平均值
- 用途：作为周结算时的建议值
- 用户拥有最终确认权

### 15.3 季度结果

- 来源：该季度所有已确认周结果的均值
- 用途：判断季度是否达标

### 15.4 季度达标判定

- 达标线：`>= 80%`
- 结果：获得 1 次季度荣誉累计

## 16. MVP 范围

MVP 必须包含：

1. 当前季度目标设置
2. 本周承诺创建
3. AI 拆解任务建议
4. Todo 任务 CRUD
5. 每日能量记录与实时修改
6. 能量球充电涌入动效
7. 周结算与果实生成
8. 季度结算与树进化
9. 季度达标后的荣誉层累积
10. 四 tab 基础结构：能量 / 清单 / 计划 / 成长树
11. 计划页时间轴（AI 协作生成 4 周计划）
12. 冷启动引导流程（一件事跑通闭环）
13. 产品人格语气系统（知性温柔大姐姐）
14. 打分前 TDL 查看提醒

MVP 暂不包含：

1. 高强度 AI 对话
2. 复杂社交分享
3. 多人协作
4. 多主题成长世界观扩展
5. 历史数据大规模回填修订

## 17. 关键风险与应对

### 风险 1：主观进度过于漂浮

应对：

- 用本周承诺做锚点
- 用 todo 完成情况做参考证据
- 周末增加一次人工确认结算

### 风险 2：AI 生成的任务不实用

应对：

- AI 仅输出建议版
- 用户始终可以 CRUD
- 用户可主动触发重新规划

### 风险 3：树很好看，但流程不成立

应对：

- 先以“季度目标 -> 本周承诺 -> 任务建议 -> 每日记录 -> 周结算 -> 季结算”流程作为 MVP 主线
- 视觉表现不先于流程规则

### 风险 4：游戏化压过产品价值

应对：

- 把树、果实、荣誉层作为反馈层，不让其替代真实规划流程
- 保持首页和清单结构克制

## 18. MVP 验证重点

MVP 上线后优先验证以下问题：

1. 用户是否真的会每周创建 3-5 条承诺
2. AI 生成的任务建议是否能减少用户规划成本
3. 用户是否愿意每天记录一次主观推进度
4. 用户是否能理解“任务完成”和“能量推进”不是同一回事
5. 周果实与季度树进化是否能形成足够强的正反馈

## 19. 本次设计结论

本产品最终确定为一款：

> 以前台极简能量交互为主舞台、以后台 AI 规划为辅助能力、以年度成长树承载长期反馈的个人成长管理 App。

它的核心闭环是：

`季度目标 -> 月计划（AI 协作） -> 本周承诺 -> AI 清单 -> 每日能量（充电动效） -> 每周果实 -> 季度进化 -> 年度成长树`

其中最关键的产品立足点有五条：

1. AI 只负责减轻规划负担，不负责做前台聊天工具。
2. 用户的主观记录必须锚定本周承诺，不能成为纯情绪滑条。
3. 长期成长反馈必须可积累、可保留、可回看。
4. 每日体验必须有即时反馈（充电动效），不能等到周末才有情绪高点。
5. 产品人格是"知性温柔的大姐姐"——不催促、不评判、始终在。

## 20. MVP 开发优先级与验证策略

### 20.1 开发原则

边开发边验证，快速迭代。不追求一次性完美，而是通过真实使用快速发现问题并调整。

### 20.2 第 1 周：核心闭环（能跑通就行）

目标：验证"季度目标 → 本周承诺 → 每日打分 → 周结算"这个流程是否顺畅。

必须完成：

1. 冷启动：输入一个季度目标（暂不做目标类型识别）
2. AI 拆解：生成本周承诺 + 任务建议
3. 能量页：展示能量球 + 今日打分滑条（暂不做充电动效）
4. 周结算：确认本周结果，生成一颗果实

验证方式：创始人自己用一周，记录体验问题。

### 20.3 第 2 周：体验打磨（让它好用）

目标：让每日体验有足够的情绪反馈，让用户愿意继续用下去。

必须完成：

1. 能量页加入"本周承诺完成概览"作为打分参照
2. 充电动效（气泡涌入 + 百分比跳动 + 颜色渐变）
3. 周结算加入仪式感（逐步回顾 + 果实生成动画）
4. 成长树加入"点击果实查看那周回顾"的时间胶囊功能

验证方式：创始人用两周后，评估是否愿意持续使用。

### 20.4 第 3 周：AI 优化（让它聪明）

目标：让 AI 生成的承诺和任务真正有用，而不是空洞的建议。

必须完成：

1. 目标类型识别（结果型 / 项目型 / 习惯型）
2. AI 拆解逻辑优化（根据目标类型生成不同的本周承诺）
3. 计划页时间轴（4 周滚动计划，AI 协作生成）
4. 产品人格语气系统（知性温柔大姐姐）

验证方式：创始人用一个月后，评估 AI 生成内容的质量。

### 20.5 关键验证指标

MVP 阶段不追求数据量，而是追求体验质量。关键验证指标：

- 创始人是否愿意每天打开 App 记录能量？
- 创始人是否觉得本周承诺的拆解有意义？
- 创始人是否觉得每日打分能反映真实推进？
- 创始人是否愿意在周末花时间做周结算？
- 创始人是否会主动打开成长树回顾过去的周？

如果这五个问题的答案都是"是"，MVP 验证通过，可以考虑扩展用户。

## 21. 原型产出

### 21.1 交互式原型

- 文件位置：`prototype/index.html`
- 类型：单文件 HTML 交互原型，可直接在浏览器中打开体验
- 视觉风格：深色系、神秘静谧、古老而充满朝气
- 设备适配：iPhone 14 Pro（393px 宽度）

### 21.2 原型覆盖页面

1. 冷启动引导流程（4 步）
   - 树苗展示 → 输入季度目标 → AI 拆解动画 → 生成本周承诺
2. 能量页（首页）
   - 能量球（带脉冲光晕动效）+ 本周累计百分比
   - 本周承诺完成概览（3 条承诺 + 进度百分比）
   - 今日推进度滑条 + 确认按钮
   - 确认后能量球颜色渐变 + 数字跳动动效
3. 清单页
   - 承诺作为可展开容器，任务嵌套在承诺内
   - 任务支持勾选完成（视觉反馈）
   - AI 重新拆解入口
4. 计划页
   - 纵向时间轴（过去周 / 当前周 / 未来周）
   - 当前周高亮，过去周显示结果，未来周虚线样式
   - AI 协作调整入口
5. 成长树页
   - CSS 绘制的树（树干 + 枝干 + 树冠 + 果实）
   - 果实可点击，弹出时间胶囊回顾卡片
   - 季度时间线（Q1-Q4 四阶段）
   - 统计卡片（累计果实 / 季度达标 / 荣誉层）
6. 周结算流程（弹窗）
   - 4 步仪式感流程：开场语 → 每日能量柱状图 → 承诺完成情况 + 调整周结果 → 果实生成动画 + 感悟输入
