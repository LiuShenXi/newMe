## 补全方案摘要

### #1 认证方案

- Access Token（JWT，15 分钟）+ Refresh Token（不透明字符串，30 天）
- Refresh Token 轮换：每次刷新旧 token 失效，检测重放则吊销全部
- MVP 采用手机号验证码登录
- 端侧 expo-secure-store 存储双令牌

### #2 API 版本策略

- URL path versioning：`/api/v1/...`
- 旧版本保留至少 2 个移动端发布周期
- 客户端 `X-App-Version` 请求头

### #3 推送通知基础设施

- Expo Notifications（封装 APNs + FCM）
- 后端 NotificationModule：令牌管理、场景调度、发送记录
- 三个推送场景：每日能量提醒、周结算引导、召回
- 推送文案遵循产品人格

### #4 端侧 SQLite 迁移

- `PRAGMA user_version` 管理 schema 版本
- `src/db/migrations/` 目录按版本号顺序执行
- 只做加列加表，不做删列
- 迁移失败降级为仅网络模式

### #5 AI 调用前端等待策略

- MVP 同步 HTTP，前端 30 秒超时，后端 25 秒超时
- 深度愿景逐层调用，单次耗时可控
- 演进条件：稳定超 10 秒时引入后台任务 + SSE

### #6 时区处理

- 服务端 UTC 存储
- 用户时区 `users.timezone`
- 自然周：周一 00:00 到周日 23:59（用户本地时间）
- MVP 不处理跨时区旅行

### #7 同步冲突解决

- 客户端携带 version，服务端自增版本号
- 服务端版本号大的赢
- 批量推送返回逐条结果
- 结算数据以首次确认为准

### #8 AI 重试与容错

- Schema 失败：重试 1 次（修复 prompt）
- 网络超时：指数退避，最多 2 次
- 熔断：10 分钟内 5 次失败，暂停 5 分钟
- MVP 不做多 Provider failover

### #9 AI 频率限制

- 同一用户同一场景：每分钟 3 次
- 同一用户全场景：每小时 30 次
- 内存计数器实现

### #10 month_goals 与 month_plans 关系

- month_goals：用户层月目标
- month_plans：执行层 4 周滚动计划容器
- 两者可独立存在，外键 `month_plans.monthGoalId` 可为 null

### #11 前后端类型共享

- monorepo pnpm workspace，共享包 `packages/shared`
- 包含 DTO 类型、枚举、常量、Zod schema
- Prisma 类型留在后端，不暴露给前端

### #12 i18n 策略

- MVP 仅支持简体中文，不做国际化

### #13 周结算触发

- 用户手动进入（能量页和成长树页提供入口）
- 周日晚推送提醒引导
- 未结算的周可补结算前一周

### #14 Deep Linking

- expo-router deep linking
- 推送通知携带路由参数跳转
- MVP 只支持 App 内路由
