# 数据与 AI 架构

- 日期：2026-04-29
- 适用范围：MVP 数据模型、同步、AI 编排

## 1. 设计目标

数据与 AI 架构需要同时满足：

1. 支撑季度目标、本周重点、今日清单、每日能量、周结算、成长树的完整闭环。
2. 保留用户修改权，AI 只生成草案，不直接替用户做决定。
3. 支持端侧弱网可用和后续云端同步。
4. 让周果实、季度荣誉和年度回看有稳定历史快照。
5. 能追踪 AI 输出质量、prompt 版本和失败原因。

## 2. 核心实体

建议服务端核心表：

| 实体 | 说明 |
| --- | --- |
| `users` | 用户身份、时区、偏好。 |
| `visions` | 五年愿景，深度规划入口。 |
| `annual_objectives` | 年度 OKR 草案确认后的正式数据。 |
| `quarters` | 年度内 Q1-Q4 时间边界和阶段。 |
| `quarter_goals` | 季度目标，含目标类型。 |
| `goal_classifications` | AI 识别的结果型、项目型、习惯型等分类记录。 |
| `month_plans` | 每次生成的 4 周滚动计划。 |
| `week_plans` | 单周计划节点。 |
| `weekly_focuses` | 本周 3-5 条重点。 |
| `todos` | 每日任务，支持 AI 建议和用户手动创建。 |
| `energy_entries` | 每日能量记录。 |
| `weekly_settlements` | 周结算最终结果和快照。 |
| `tree_fruits` | 每周果实及时间胶囊。 |
| `quarter_honors` | 季度达标产生的永久荣誉层。 |
| `ai_generations` | AI 生成记录、prompt 版本、输出、状态。 |
| `sync_devices` | 端侧设备和同步水位。 |
| `sync_changes` | 可选的变更日志，后续增强同步时启用。 |

## 3. 实体关系

```text
User
  ├─ Vision
  ├─ AnnualObjective
  │   └─ QuarterGoal
  │       └─ WeekPlan
  │           ├─ WeeklyFocus
  │           └─ Todo
  ├─ EnergyEntry
  ├─ WeeklySettlement
  │   └─ TreeFruit
  └─ QuarterHonor
```

关键关系：

1. 一个用户一年可以有一个主年度规划版本，后续可扩展多版本。
2. 一个季度包含多个季度目标。
3. 一个周计划包含 3-5 条本周重点。
4. Todo 归属于具体日期，也可关联来源本周重点。
5. EnergyEntry 归属于具体日期和自然周。
6. WeeklySettlement 记录结算时的快照，不只引用实时数据。
7. TreeFruit 从 WeeklySettlement 派生，但保留独立展示字段。

## 4. 本地数据子集

端侧本地不需要完整保存所有后端表，但要覆盖高频体验：

1. 当前愿景和已确认规划摘要。
2. 当前季度目标。
3. 最近 4 周计划。
4. 当前周重点。
5. 本周 7 天任务。
6. 每日能量记录。
7. 已确认周结算和果实。
8. 当前年度树阶段和荣誉层。
9. AI 草案缓存和同步队列。

端侧不建议长期保存完整 AI prompt、模型响应元数据和调试日志，这些由后端受控保存。

## 5. 关键字段规则

所有核心业务表建议保留：

1. `id`
2. `userId`
3. `createdAt`
4. `updatedAt`
5. `deletedAt`
6. `source`：user、ai、system。
7. `version`

AI 建议类数据额外保留：

1. `aiGenerationId`
2. `promptVersion`
3. `userEdited`
4. `invalidatedAt`

结算快照类数据额外保留：

1. `snapshotJson`
2. `finalScore`
3. `suggestedScore`
4. `confirmedAt`
5. `reflection`

## 6. 同步策略

MVP 同步策略保持简单：

1. 用户操作先写本地。
2. 本地记录进入 `sync_queue`。
3. 网络可用时批量推送变更。
4. 服务端返回远端 ID、版本和更新时间。
5. 端侧按水位拉取远端变更。
6. 冲突先采用最后修改时间和服务端版本处理。

冲突原则：

1. 同一设备内本地用户操作优先展示。
2. 不允许 AI 结果自动覆盖用户已编辑内容。
3. 周结算确认后视为历史快照，不参与实时合并。
4. 删除为软删除，服务端保留墓碑记录。
5. MVP 先以单设备为主，多设备复杂冲突作为后续演进。

## 7. 计算规则

### 7.1 本周累计进度

来源：本周已记录的每日能量值。

默认计算：

```text
本周累计 = 已记录每日能量之和 / 已记录天数
```

如产品后续希望按 7 天完整周计算，可在产品文档和后端计算规则中同步修改。

### 7.2 建议周结果

周结算时由系统给出建议值：

```text
建议周结果 = 本周已记录每日能量平均值
```

用户拥有最终确认权。最终周结果写入 `weekly_settlements.finalScore`。

### 7.3 季度结果

```text
季度结果 = 该季度已确认周结果平均值
```

季度均值 `>= 80%` 时新增一次季度荣誉。

### 7.4 树阶段

树阶段由时间所属季度决定：

1. Q1：起始阶段。
2. Q2：成长阶段。
3. Q3：丰茂阶段。
4. Q4：年度完成阶段。

树阶段不因季度未达标而停滞。

## 8. AI 场景

MVP AI 场景：

| 场景 | 输入 | 输出 |
| --- | --- | --- |
| 快速季度规划 | 季度目标文本 | 目标类型、本周重点、今日清单建议 |
| 愿景到年度 OKR | 五年愿景 | 年度 OKR 草案 |
| 年度到季度 OKR | 年度 OKR | 4 个季度 OKR 草案 |
| 季度到 4 周承诺 | 当前季度 OKR | 首月 4 周承诺 |
| 本周重点到清单 | 季度目标、本周重点、已有任务 | 7 天任务建议，默认突出今日 |
| 后续周重规划 | 当前计划、用户反馈、本周结果 | 后续周计划调整建议 |

AI 不做：

1. 常驻聊天。
2. 自动替用户打能量分。
3. 自动覆盖用户任务。
4. 未经确认直接改变正式计划。

## 9. AI 编排流程

```text
业务模块构造上下文
  -> AiModule 选择场景和 prompt 版本
  -> 调用模型供应商
  -> 解析 JSON
  -> Schema 校验
  -> 业务规则校验
  -> 保存 ai_generations
  -> 返回草案
```

确认流程：

```text
用户编辑草案
  -> 提交确认
  -> 校验上下文版本
  -> 业务模块事务落库
  -> ai_generations 标记 confirmed
```

重新生成流程：

```text
用户点击重新生成
  -> 传入 regenerateFromId
  -> 只生成当前层级草案
  -> 不影响下游已确认数据
```

## 10. 结构化输出

AI 输出必须使用 JSON Schema 或 Zod schema 校验。示例结构：

```json
{
  "goalType": "project",
  "weeklyFocuses": [
    {
      "title": "完成登录模块前后端闭环",
      "reason": "这是本周推进项目可用性的关键结果"
    }
  ],
  "todayTodos": [
    {
      "title": "梳理登录接口字段",
      "estimatedMinutes": 30,
      "sourceFocusTitle": "完成登录模块前后端闭环"
    }
  ]
}
```

校验失败处理：

1. 自动重试一次，使用更严格的修复 prompt。
2. 仍失败则返回可重试错误。
3. 记录失败原因和原始输出摘要。
4. 不把失败输出写入正式业务表。

## 11. Prompt 版本管理

每个 AI 场景需要有独立 prompt 版本：

```text
quick_quarter_plan:v1
vision_to_annual_okr:v1
annual_to_quarter_okr:v1
quarter_to_four_week_commitments:v1
weekly_focus_to_todos:v1
replan_future_weeks:v1
```

`ai_generations` 记录：

1. `scenario`
2. `promptVersion`
3. `model`
4. `inputHash`
5. `outputJson`
6. `status`
7. `errorCode`
8. `latencyMs`
9. `tokenUsage`
10. `confirmedAt`

## 12. 成本与隐私

成本控制：

1. 只在用户主动触发规划或重新规划时调用 AI。
2. 对相同上下文可缓存短期结果。
3. 限制每个草案层级的重新生成频率。
4. 记录 token 使用，按场景观察成本。

隐私原则：

1. 端侧不保存模型 API Key。
2. 日志不直接打印完整愿景、目标和感悟正文。
3. AI 生成记录只保存必要排查信息。
4. 后续如支持导出或删除账号，需要能删除用户个人内容和 AI 记录。
