# 前端架构设计

- 日期：2026-04-29
- 技术栈：Flutter + Riverpod + SQLite/Drift
- 适用范围：移动端 App MVP

## 1. 前端架构目标

前端是本产品体验成败的主战场，需要同时满足：

1. 每日记录路径足够短，用户能快速完成今日清单查看和能量记录。
2. 冷启动深度规划可返回、可重新生成、可修改，不丢状态。
3. 能量球、滑条、果实、成长树和周结算动效稳定流畅。
4. 弱网下今日清单、本周重点和每日能量仍可使用。
5. 四 tab 结构清晰，不让 AI、统计和复杂任务列表压过主舞台。

## 2. 工程分层

建议采用 feature-first 目录结构：

```text
lib/
  app/
    app.dart
    router.dart
    theme.dart
  shared/
    widgets/
    design_tokens/
    utils/
    errors/
  core/
    database/
    network/
    sync/
    analytics/
  features/
    onboarding/
    energy/
    todo/
    plan/
    tree/
    settlement/
    goal/
```

每个 feature 内部保持同样分层：

```text
feature/
  presentation/  页面、组件、动画
  application/   Riverpod notifier、use case
  domain/        实体、值对象、业务规则
  data/          repository、DAO、API client
```

## 3. 导航结构

主 App 固定为 4 个 tab：

1. `能量`：首页，高频入口。
2. `清单`：今日 todolist 和本周 7 天概览。
3. `计划`：月计划按周、年计划按 Q。
4. `成长树`：年度树、果实时间胶囊、荣誉层。

特殊流程：

1. 冷启动规划作为首次进入的全屏流程，不挂在底部 tab 下。
2. 周结算作为独立流程，可从能量页、成长树页或演示入口进入。
3. 本周概览可作为清单页内的右侧滑入面板或独立子路由。
4. 计划页点击周节点跳转到清单页，并直接打开对应周概览。

路由设计需要支持带参数进入，例如：

```text
/todos?weekId=current&source=plan
/settlement/week/:weekId
/tree/fruits/:fruitId
```

## 4. 状态管理

Riverpod 使用原则：

1. 页面临时态使用 `StateProvider` 或局部 `StatefulWidget`。
2. 业务状态使用 `Notifier` 或 `AsyncNotifier`。
3. 数据读取通过 repository provider 注入。
4. 本地数据库查询优先暴露 Stream，页面自动响应变更。
5. AI 生成、同步、结算提交等异步动作需要显式 loading、success、error 状态。

关键状态对象：

| 状态 | 说明 |
| --- | --- |
| `CurrentWeekProvider` | 当前自然周、周起止日期、当前季度归属。 |
| `OnboardingFlowProvider` | 冷启动路径、步骤、输入、AI 草案、返回栈。 |
| `WeeklyFocusProvider` | 本周重点列表和变更版本。 |
| `TodayTodoProvider` | 今日任务列表、完成数、编辑态。 |
| `WeeklyTodoOverviewProvider` | 本周 7 天任务聚合。 |
| `EnergyEntryProvider` | 今日能量值、本周累计、打分前提醒状态。 |
| `PlanTimelineProvider` | 月计划和年计划视图数据。 |
| `TreeSnapshotProvider` | 树阶段、果实、荣誉层。 |
| `SyncStatusProvider` | 本地脏数据、同步进度、失败原因。 |

## 5. 本地数据层

端侧使用 Drift 管理 SQLite。建议本地表覆盖：

1. `local_goals`
2. `local_weekly_focuses`
3. `local_todos`
4. `local_energy_entries`
5. `local_weekly_settlements`
6. `local_tree_fruits`
7. `local_quarter_honors`
8. `local_ai_drafts`
9. `sync_queue`

每条可同步记录保留：

1. `id`：本地 UUID。
2. `remoteId`：服务端 ID，可为空。
3. `createdAt`、`updatedAt`。
4. `deletedAt`：软删除。
5. `syncStatus`：pending、synced、failed。
6. `version`：服务端版本或本地递增版本。

写入策略：

1. 用户操作先写本地，保证界面立即反馈。
2. 同步队列异步推送到后端。
3. AI 生成依赖网络，成功后写入本地草案；用户确认后再转为正式数据。
4. 周结算确认后生成快照，避免历史受后续编辑影响。

## 6. 页面模块设计

### 6.1 冷启动规划

冷启动需要支持两条路径：

1. 快速规划：季度目标 -> AI 拆解 -> 本周重点 + 今日清单建议。
2. 深度愿景：五年愿景 -> 年度 OKR -> 4 个季度 OKR -> 首月 4 周承诺 -> 天计划。

关键交互规则：

1. 返回上一级只负责导航，不清空输入，不触发重新生成。
2. 重新生成只更新当前草案层级。
3. 确认后进入下一层，已确认内容保持稳定。
4. 用户修改过的 AI 草案必须标记为 userEdited，避免后续被无意覆盖。

### 6.2 能量页

能量页只做两件事：

1. 展示本周推进状态。
2. 记录并修改今日推进度。

组件建议：

1. `EnergyOrb`：能量球，自绘或组合动画。
2. `WeeklyFocusProgressPanel`：本周重点推进概览，视觉降级。
3. `EnergySlider`：今日推进度滑条，支持发光轨道和粒子效果。
4. `ConfirmEnergyButton`：确认今日能量，未查看清单时触发温和提醒。

动画要求：

1. 能量确认后播放 1-2 秒充电涌入动效。
2. 百分比数字跟随动画递增。
3. 低性能设备允许降低粒子数量。
4. 页面进入时不自动播放强动效，避免打扰。

### 6.3 清单页

清单页默认展示今日 todolist。

功能边界：

1. 支持新增、编辑、勾选、左滑删除。
2. 右上角进入本周 7 天概览。
3. 顶部轻量展示本周重点摘要。
4. AI 任务只是建议来源，用户始终可改。
5. 本周重点变更后，旧 AI 建议标记失效，但不自动覆盖任务。

### 6.4 计划页

计划页是中频视图。

结构：

1. 顶部切换 `月计划 · 按周` 和 `年计划 · 按 Q`。
2. 月计划展示最近 4 周，当前周高亮。
3. 年计划展示 Q1-Q4 的目标、阶段说明和进度。
4. AI 重规划需要用户先输入变化反馈，最多 3 轮迭代。
5. 点击周节点可跳到清单页本周概览。

### 6.5 成长树页

成长树是长期反馈视图。

实现建议：

1. 树主体用 CustomPainter 或分层 Widget 实现。
2. 果实是可点击对象，点击打开时间胶囊。
3. 树阶段由季度决定。
4. 荣誉层由年度累计达标次数决定，只增不减。
5. 复杂统计不要压在树主画面上，通过详情入口展示。

### 6.6 周结算

周结算是独立流程：

1. 开场语。
2. 每日能量回顾。
3. 本周重点推进回顾。
4. 建议周结果。
5. 用户微调最终结果。
6. 周感悟。
7. 果实生成动画。

提交前需要构造结算快照，包含当时的每日能量、重点、任务摘要和用户感悟。

## 7. 错误与空状态

前端错误处理原则：

1. AI 失败时保留用户输入，允许重试，不清空流程。
2. 同步失败时本地数据继续可见，并在状态入口提示。
3. 删除任务后提供短暂撤销机会。
4. 周结算提交失败时保留已填写内容。
5. 网络错误使用温和语气，不使用惩罚式文案。

典型空状态：

1. 今日无任务：允许手动添加，也可触发 AI 建议。
2. 本周无重点：引导去计划页或冷启动结果确认。
3. 成长树无果实：展示当前树苗，不制造失败感。
4. AI 生成中：展示轻量生成状态，不伪装聊天机器人。

## 8. 测试策略

前端测试分层：

1. Domain 单元测试：能量均值、季度达标、结算快照生成。
2. Repository 测试：本地写入、软删除、同步状态。
3. Widget 测试：冷启动返回、重新生成入口、清单 CRUD、能量确认提醒。
4. Golden 测试：核心页面在主尺寸下的视觉稳定性。
5. Integration 测试：快速规划到能量记录、周结算到果实生成。

静态原型仍使用 `npx playwright` 做视觉和交互回归；Flutter App 进入实现后再补充 Flutter 自身测试链路。
