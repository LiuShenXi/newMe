# 技术选型决策

- 日期：2026-04-29
- 决策状态：MVP 阶段采用
- 适用范围：个人成长管理 App 首个可用版本

## 1. 背景

产品核心是个人成长闭环，不是重项目管理工具，也不是 AI 聊天助理。技术选型需要优先支持：

1. Android 和 iOS 双端可用。
2. 能量球、果实生成、成长树等强视觉反馈。
3. 今日清单、每日能量、本周重点等核心数据弱网可用。
4. AI 在规划和重规划节点稳定输出结构化结果。
5. 创始人能以较低学习成本持续迭代。
6. MVP 阶段不过早承担微服务、三端原生、复杂基础设施。

## 2. 最终选型

| 层级 | 选型 | 说明 |
| --- | --- | --- |
| 移动端框架 | Flutter | 一套代码覆盖 Android/iOS，动画和自绘能力适合能量球、果实、成长树。 |
| 移动端语言 | Dart | Flutter 标准语言，声明式 UI 心智接近前端。 |
| 状态管理 | Riverpod | 比 Bloc 更轻，适合 MVP 到中等复杂度业务状态。 |
| 本地数据库 | SQLite + Drift | 适合结构化离线数据、类型安全查询和本地数据流。 |
| 后端框架 | NestJS | 模块化清晰，创始人有基础，适合 AI 编排和业务快速迭代。 |
| 后端语言 | TypeScript | DTO、Schema、AI 结构化输出和全栈类型协作更顺手。 |
| 主数据库 | PostgreSQL | 稳定可靠，适合关系型业务、统计和长期演进。 |
| ORM | Prisma | TypeScript 体验好，MVP 开发效率高。 |
| AI 接入 | 后端统一封装 | 端侧不直连模型，便于密钥保护、成本控制、prompt 版本和模型切换。 |
| 部署方式 | Docker 单体 | 首期简单可控，后续按压力拆分队列或 worker。 |

## 3. 前端选型

### 3.1 采用 Flutter

采用 Flutter 的主要原因：

1. 双端一致性好，适合移动 App 首发。
2. 动画、自绘、Canvas 能力强，适合能量球、成长树和果实动效。
3. UI 不依赖原生控件风格，能稳定承载产品的深色玻璃质感和青绿色能量反馈。
4. 后续可以用同一套领域和数据层扩展平板或桌面预览。

### 3.2 采用 Riverpod

Riverpod 用于管理：

1. 页面局部状态，例如 Tab、弹层、编辑态。
2. 异步状态，例如 AI 草案生成、远端同步、结算提交。
3. 领域状态，例如当前周、今日清单、本周重点、成长树快照。
4. Repository 和 use case 依赖注入。

不采用 Bloc 的原因是首期业务复杂度还不需要较重事件流样板。后续如果某些复杂流程需要事件溯源，可以在局部模块引入更明确的状态机，而不是全局切换。

### 3.3 采用 SQLite + Drift

端侧需要支持弱网和低成本记录，SQLite + Drift 负责：

1. 本周重点、今日清单、每日能量、本周累计的本地读写。
2. 周结算和成长树历史快照的本地查看。
3. 同步水位、脏数据标记、软删除记录。

不直接只依赖后端的原因是每日记录场景很轻，如果网络抖动就阻断体验，会破坏产品的低脑力原则。

## 4. 后端选型

### 4.1 采用 NestJS

NestJS 适合本产品的原因：

1. 模块边界天然清晰，适合拆分目标、计划、清单、能量、结算、成长树和 AI。
2. TypeScript 对结构化 AI 输出、DTO 校验和接口类型管理友好。
3. 创始人已有 NestJS 基础，能降低启动成本。
4. 单体部署足够支撑 MVP，后续也能按模块拆出 worker 或独立服务。

### 4.2 采用 PostgreSQL + Prisma

PostgreSQL 承担长期可靠存储，Prisma 提供类型安全访问。

适合的原因：

1. 数据关系明确：目标、周重点、任务、能量、结算、果实之间存在稳定关联。
2. 周结算、季度结算和年度回看需要可靠聚合。
3. AI 生成记录、prompt 版本和业务快照需要长期留痕。
4. Prisma 能在 MVP 阶段提升建模和迁移效率。

### 4.3 Go 的定位

MVP 阶段不采用 Go 作为主后端。Go 可作为后续候选，适合：

1. 高吞吐后台 worker。
2. 独立推送服务。
3. 大量统计或批处理。
4. 与 NestJS 主业务解耦的性能型服务。

当前阶段切换 Go 的收益小于学习和协作成本。

## 5. AI 选型原则

AI 不作为端侧 SDK 接入，统一由后端封装。

后端负责：

1. 管理模型供应商和密钥。
2. 管理 prompt 模板和版本。
3. 控制 AI 调用频率和成本。
4. 校验结构化输出。
5. 记录输入摘要、输出结果、失败原因和重试次数。
6. 允许后续切换模型，不影响端侧协议。

MVP 的 AI 场景只覆盖规划和重规划，不做常驻聊天。

## 6. 鸿蒙兼容策略

MVP 对华为设备的目标是“能安装和使用”，不是 HarmonyOS NEXT 原生首发。

执行策略：

1. 首期通过 Flutter Android 包覆盖华为 Android 兼容设备。
2. 不在 MVP 主线投入 ArkTS/ArkUI。
3. 不承诺 HarmonyOS NEXT 原生应用上架。
4. 当华为分发成为关键增长点时，再做专项 Spike。

专项验证内容：

1. Flutter Android 包在目标华为设备上的安装、权限、通知和存储表现。
2. Flutter OHOS 插件兼容性、构建签名、运行稳定性。
3. ArkTS/ArkUI 重写关键页面的成本和体验收益。

## 7. 暂不采用方案

| 方案 | 暂不采用原因 |
| --- | --- |
| React Native | 动画、自绘和跨端视觉一致性不如 Flutter 符合本产品重点。 |
| Android/iOS/鸿蒙三端原生 | 人力成本过高，MVP 阶段体验闭环还需验证。 |
| 微服务 | 当前没有高并发或团队边界需求，会增加部署和调试成本。 |
| Kubernetes | 单体 Docker 已能覆盖首期部署需求。 |
| Redis/消息队列 | 首期 AI 和结算量可同步处理，等耗时和排队真实出现后再引入。 |
| 端侧直连 AI | 密钥、成本、prompt 版本和输出校验都不可控。 |

## 8. 当前结论

MVP 技术栈确定为：

```text
Flutter + Riverpod + SQLite/Drift
NestJS + TypeScript + PostgreSQL + Prisma
后端统一封装 AI 能力
Docker 单体部署
```

这套方案优先服务于快速验证产品闭环、降低创始人迭代成本、保留后续扩展空间。
