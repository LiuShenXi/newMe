# 后端架构设计

- 日期：2026-04-29
- 技术栈：NestJS + TypeScript + PostgreSQL + Prisma
- 部署形态：MVP 模块化单体

## 1. 后端架构目标

后端负责四类能力：

1. 可靠保存目标、计划、清单、能量、结算和成长树数据。
2. 封装 AI 生成能力，保证 prompt、密钥、模型和结构化校验不暴露给端侧。
3. 支持端侧弱网和后续同步，避免移动端体验完全依赖实时网络。
4. 为周结算、季度结算、长期回看提供稳定业务规则和历史快照。

MVP 采用模块化单体，先保持部署简单，代码内部按业务边界隔离。

## 2. 工程结构

建议结构：

```text
src/
  main.ts
  app.module.ts
  common/
    decorators/
    filters/
    guards/
    interceptors/
    pipes/
    errors/
  config/
  prisma/
  modules/
    auth/
    users/
    goals/
    plans/
    todos/
    energy/
    settlements/
    tree/
    ai/
    sync/
```

每个业务模块内部保持：

```text
module/
  module.ts
  controller.ts
  service.ts
  dto/
  entities-or-types/
  repository.ts
  policy.ts
  tests/
```

## 3. 模块边界

| 模块 | 职责 | 不负责 |
| --- | --- | --- |
| `AuthModule` | 登录、令牌刷新、用户身份上下文 | 业务权限细节 |
| `UsersModule` | 用户资料、偏好、时区 | 目标和计划内容 |
| `GoalsModule` | 五年愿景、年度 OKR、季度目标、目标类型 | 每日任务 CRUD |
| `PlansModule` | 首月承诺、月计划、本周重点、重规划版本 | AI 模型直接调用 |
| `TodosModule` | 今日任务、本周任务概览、任务 CRUD | 每日能量分数计算 |
| `EnergyModule` | 每日能量记录、本周累计、查看清单提醒状态 | 周结算最终确认 |
| `SettlementsModule` | 周结算、季度结算、结算快照 | 成长树渲染细节 |
| `TreeModule` | 果实、时间胶囊、季度阶段、荣誉层 | 结算均值计算入口 |
| `AiModule` | AI 场景编排、prompt、调用、校验、记录 | 直接落正式业务数据 |
| `SyncModule` | 端侧同步、变更水位、冲突处理 | 业务规则本身 |

边界原则：

1. AI 只产出草案，业务模块负责确认和落库。
2. 周结算通过 Settlement 创建快照，再通知 Tree 生成果实。
3. Todo 的完成情况可作为 Energy 的参考数据，但不直接决定能量分数。
4. Plan 中的本周重点是 Todo 和 Energy 的上游锚点。

## 4. API 形态

MVP 默认采用 REST + JSON。AI 生成若耗时明显，可扩展为任务轮询或 SSE。

核心接口示例：

| 场景 | 方法 | 路径 | 说明 |
| --- | --- | --- | --- |
| 获取当前用户上下文 | GET | `/me` | 用户、时区、当前周信息 |
| 创建/更新愿景 | PUT | `/goals/vision` | 深度规划起点 |
| 设置季度目标 | POST | `/goals/quarters/:quarterId/goals` | 快速规划或季度规划 |
| 生成 AI 草案 | POST | `/ai/generations` | 按场景生成结构化草案 |
| 确认 AI 草案 | POST | `/ai/generations/:id/confirm` | 转为正式业务数据 |
| 获取本周重点 | GET | `/plans/weeks/:weekId/focuses` | 清单和能量共同使用 |
| 更新本周重点 | PUT | `/plans/weeks/:weekId/focuses` | 标记旧 AI 建议失效 |
| 获取今日任务 | GET | `/todos/today` | 今日清单 |
| 修改任务 | POST/PATCH/DELETE | `/todos` | CRUD 和软删除 |
| 记录今日能量 | PUT | `/energy/days/:date` | 当天可多次修改 |
| 获取本周能量 | GET | `/energy/weeks/:weekId` | 本周累计和日记录 |
| 创建周结算 | POST | `/settlements/weeks/:weekId` | 生成周快照和果实 |
| 获取成长树 | GET | `/tree/years/:year` | 树阶段、果实、荣誉 |
| 同步变更 | POST | `/sync/push` | 端侧脏数据推送 |
| 拉取变更 | GET | `/sync/pull` | 按水位拉取远端变更 |

## 5. AI 生成接口设计

统一入口：

```text
POST /ai/generations
```

请求核心字段：

```json
{
  "scenario": "quick_quarter_plan",
  "input": {},
  "contextVersion": "string",
  "regenerateFromId": "optional-generation-id"
}
```

`scenario` 建议枚举：

1. `quick_quarter_plan`
2. `vision_to_annual_okr`
3. `annual_to_quarter_okr`
4. `quarter_to_four_week_commitments`
5. `weekly_focus_to_todos`
6. `replan_future_weeks`

响应只返回草案和生成记录 ID，不直接覆盖正式数据。

确认接口由业务模块参与事务：

```text
POST /ai/generations/:id/confirm
```

确认时需要：

1. 校验草案仍匹配当前上下文版本。
2. 合并用户编辑后的内容。
3. 写入正式业务表。
4. 标记生成记录为 confirmed。

## 6. 事务与一致性

需要事务保护的场景：

1. 确认冷启动规划结果，同时创建目标、本周重点、今日任务。
2. 更新本周重点，并标记旧 AI 任务建议失效。
3. 创建周结算，同时生成果实和树时间胶囊。
4. 季度结算，同时更新树阶段和荣誉层。
5. 同步端侧批量变更。

一致性规则：

1. 周结果确认后不因任务后续编辑自动变化。
2. 季度荣誉只增不减。
3. AI 重新生成只更新当前草案层级。
4. 用户手动编辑优先于 AI 草案。
5. 删除默认软删除，避免同步时丢历史。

## 7. 鉴权与权限

MVP 可从单用户体验起步，但后端仍需要保留用户隔离：

1. 所有业务表带 `userId`。
2. Controller 只从认证上下文取用户 ID，不信任客户端传入的 `userId`。
3. 查询必须限定当前用户。
4. AI 生成记录归属当前用户。
5. 同步接口按用户和设备维度维护水位。

登录方式可以先采用邮箱验证码、Apple 登录或本地开发 mock，具体由产品发布策略决定。

## 8. 错误处理

统一错误结构：

```json
{
  "code": "AI_OUTPUT_INVALID",
  "message": "生成结果暂时不可用，请稍后重试",
  "requestId": "req_xxx",
  "details": {}
}
```

错误分类：

1. `VALIDATION_ERROR`：请求参数不合法。
2. `AUTH_REQUIRED`：未登录或令牌失效。
3. `RESOURCE_NOT_FOUND`：资源不存在或不属于当前用户。
4. `CONFLICT`：上下文版本冲突。
5. `AI_OUTPUT_INVALID`：AI 输出结构不合格。
6. `AI_PROVIDER_ERROR`：模型供应商失败。
7. `SYNC_CONFLICT`：同步冲突。

前端展示使用温和文案，后端日志保留技术细节。

## 9. 可观测性

首期至少记录：

1. `requestId`
2. 用户 ID 的脱敏标识
3. 接口路径、耗时、状态码
4. AI 场景、prompt 版本、模型、token 估算、重试次数
5. 同步批次大小和失败原因
6. 周结算和季度结算任务结果

不在普通日志中记录完整愿景、目标和感悟正文。需要排查 AI 质量时，可在受控的 AI 生成记录中保存必要摘要和输出。

## 10. 测试策略

后端测试分层：

1. 单元测试：目标类型识别结果处理、能量均值、季度达标、荣誉层累积。
2. Service 测试：周结算事务、AI 草案确认、本周重点变更。
3. API 测试：鉴权、DTO 校验、错误结构。
4. Repository 测试：Prisma 查询、软删除、用户隔离。
5. 集成测试：冷启动快速规划确认、深度规划确认、端侧同步批处理。

AI 测试需要使用固定 mock 输出，避免模型不稳定导致测试飘动。
