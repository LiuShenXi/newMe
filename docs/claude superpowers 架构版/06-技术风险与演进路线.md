# 技术风险与演进路线

- 日期：2026-04-29
- 适用范围：MVP 到体验稳定阶段

## 1. 风险评估原则

本产品的主要风险不在高并发，而在体验闭环是否成立、AI 输出是否有用、每日记录是否足够轻、长期反馈是否能持续。技术风险评估需要优先关注这些点，而不是提前堆复杂基础设施。

## 2. 关键风险矩阵

| 风险 | 影响 | 概率 | 优先级 | 应对 |
| --- | --- | --- | --- | --- |
| AI 生成内容空泛 | 用户不信任规划结果 | 高 | 高 | 目标类型识别、结构化输出、用户可编辑、生成记录评估 |
| 主观能量分数漂浮 | 用户觉得记录无意义 | 中 | 高 | 能量页展示本周重点和清单完成作为参照 |
| 冷启动三路径选择困难 | 首次使用流失 | 中 | 高 | 推荐权重给深度路径，手动路径文案强调"自主安排" |
| 手动路径空层级体验 | 计划页显得空洞 | 中 | 中 | 空层级有补填入口，不阻断核心闭环 |
| 端侧同步冲突 | 数据错乱或丢失 | 中 | 中 | MVP 单设备优先、软删除、版本号、结算快照 |
| RN Skia 动画卡顿 | 核心反馈体验受损 | 中 | 中 | 自绘性能测试、粒子降级、低性能模式 |
| 两周上架时间压力 | 功能不完整 | 高 | 高 | 严格按 MVP 范围，先跑通闭环再打磨 |
| 过度设计拖慢 MVP | 开发周期拉长 | 高 | 高 | 单体部署、暂不引入队列和微服务 |
| 数据备份不足 | 真实使用数据丢失 | 低 | 高 | 每日备份、恢复演练、迁移前备份 |
| 隐私和日志泄露 | 用户信任受损 | 中 | 高 | 日志脱敏、AI 记录受控、密钥不入库 |
| 周结算历史被实时数据污染 | 果实和回看失真 | 中 | 高 | 周结算写入不可变快照 |

## 3. AI 风险

### 3.1 输出质量不稳定

风险表现：

1. 本周重点过泛，不能指导行动。
2. 今日任务过大或过碎。
3. 不同目标类型使用同一种拆解方式。
4. 重新规划覆盖了用户原本想保留的内容。
5. 手动路径局部辅助生成的内容与用户已有手动内容不协调。

应对：

1. 先做目标类型识别：结果型、项目型、习惯型。
2. 每个 AI 场景使用独立 prompt 和 JSON Schema。
3. 输出必须经过业务规则校验。
4. 用户确认前只作为草案。
5. 记录 prompt 版本、输出和用户是否编辑，用于后续评估。
6. 局部辅助时传入用户已有手动内容作为上下文，避免冲突。

### 3.2 成本不可控

风险表现：

1. 用户频繁重新生成。
2. 深度规划层级多，单次冷启动成本高。
3. prompt 上下文过长。

应对：

1. AI 只在关键节点由用户主动触发。
2. 重新生成设置合理频率限制。
3. 上下文只传必要摘要，不传完整历史。
4. 记录 token 使用，按场景观察成本。
5. 后续可对相同上下文做短期缓存。

## 4. 数据风险

### 4.1 本地与云端冲突

MVP 先按单设备优先处理。

首期策略：

1. 每条记录保留 `updatedAt`、`version`、`deletedAt`。
2. 用户本地操作立即可见。
3. 服务端按用户和设备维护同步水位。
4. 软删除替代物理删除。
5. 结算数据一旦确认，不参与普通实时合并。

升级条件：用户开始明显多设备使用，同一条数据频繁在多端编辑。

### 4.2 历史快照漂移

风险：用户后续编辑任务或重点，导致过去周果实解释变化。

应对：

1. 周结算写入 `snapshotJson`。
2. 果实时间胶囊读取快照，而不是实时任务表。
3. 周结果确认后只能追加备注，不自动重算历史。

## 5. 前端体验风险

### 5.1 动画性能

能量球、滑条粒子、成长树和果实是核心体验，但不能牺牲流畅性。

应对：

1. 动画组件和业务组件分离。
2. 粒子数量按设备性能降级。
3. 列表和树详情避免与重动画同帧大量重建。
4. 使用 React Native performance monitor 和真机测试。
5. 首期先实现关键反馈，不追求复杂特效堆叠。

### 5.2 首页信息过载

风险：首页被任务、统计、AI 入口塞满，破坏每日记录心智。

应对：

1. 能量页只保留能量球、本周重点概览、今日滑条、确认按钮。
2. AI 不出现在首页常驻入口。
3. 复杂统计放到计划页、成长树详情或后续分析页面。

## 6. 后端与运维风险

### 6.1 单体膨胀

风险：模块边界不清，后续维护困难。

应对：

1. NestJS 模块按业务边界拆。
2. AI 编排不直接写正式业务表。
3. 周结算和树生成通过清晰 service 接口协作。
4. 当模块过大时先做内部拆分，不急于拆服务。

### 6.2 部署和数据恢复薄弱

风险：真实使用后数据不可恢复。

应对：

1. production 每日备份。
2. 迁移前额外备份。
3. 每月做恢复演练。
4. 健康检查和错误日志上线前就具备。

## 7. 阶段路线

### 阶段 0：原型和架构基线（当前）

目标：保持产品、原型和技术架构一致。

完成标准：

1. 静态原型覆盖核心流程。
2. 产品需求文档和架构文档同步。
3. 技术选型确定。
4. MVP 模块边界明确。

### 阶段 1：MVP 可用版（第 1-2 周）

目标：跑通从规划到周结算的最小闭环。

技术重点：

1. React Native + Expo 四 tab 基础 App。
2. 冷启动三路径（深度愿景、快速规划、手动 OKR）。
3. 本地 expo-sqlite 数据层。
4. NestJS 单体 API。
5. PostgreSQL 核心模型。
6. AI 拆解接口（模型无关）。
7. 周结算和成长树果实。

暂不追求：

1. 多设备复杂同步。
2. 精细运营后台。
3. 微服务和队列。

### 阶段 2：体验稳定版（第 3-4 周）

目标：让创始人和小范围用户愿意持续使用。

技术重点：

1. 能量球和成长树动画性能优化。
2. AI 生成质量评估和 prompt 迭代。
3. 同步失败恢复。
4. 错误上报和崩溃监控。
5. 推送提醒和召回文案。
6. 更完整的周结算体验。

### 阶段 3：扩展验证版（后续）

目标：根据真实使用数据决定扩展方向。

可能方向：

1. 多端同步增强。
2. 数据导出和年度报告。
3. AI 拆解质量评分体系。
4. Redis/队列/worker。
5. RN 鸿蒙兼容验证。
6. 轻量后台和运营工具。

## 8. 技术决策门槛

| 决策 | 触发门槛 |
| --- | --- |
| 引入 Redis/队列 | AI、推送、结算出现明显排队或重试需求 |
| 拆分 worker | 后台任务影响 API 响应或部署节奏 |
| 拆微服务 | 模块团队边界或独立扩缩容需求真实出现 |
| 做多设备高级同步 | 用户反馈跨设备冲突影响体验 |
| 做 HarmonyOS NEXT | 华为分发成为明确增长或商业要求 |
| 做后台管理系统 | 出现运营、客服或 AI 质量人工审核需求 |

## 9. 收口原则

每次技术迭代结束时需要检查：

1. 实现是否仍符合产品需求文档。
2. 原型是否仍能表达当前核心体验。
3. 架构文档是否需要同步更新。
4. 是否引入了未记录的新技术依赖。
5. 是否改变了 AI、数据或结算规则。
6. 是否存在需要新增测试或运维检查的风险。

如果实现和文档不一致，以当前真实实现为准修正文档，并明确后续是否需要回到原规划。
